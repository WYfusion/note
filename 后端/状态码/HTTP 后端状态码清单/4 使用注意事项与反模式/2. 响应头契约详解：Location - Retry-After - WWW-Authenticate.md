## 概述

HTTP 状态码不是孤立的——某些状态码**必须或应该**配合特定的响应头才能完整表达语义。本页详解三个最关键的响应头契约。

---

## 1. Location 头

### 适用状态码

|状态码|Location 作用|是否必须|
|---|---|---|
|201 Created|指向新创建的资源 URI|RFC **推荐**|
|301/308|指向永久新地址|**必须**|
|302/303/307|指向临时跳转地址|**必须**|
|202 Accepted|可指向任务状态查询 URL|可选但推荐|

### 201 + Location 示例

```Python
from fastapi import FastAPI
from fastapi.responses import JSONResponse

@app.post("/users", status_code=201)
async def create_user(data: UserCreate):
    user = await user_service.create(data)
    return JSONResponse(
        status_code=201,
        content={"id": user.id, "name": user.name},
        headers={"Location": f"/api/v1/users/{user.id}"}
    )
```

> [!tip] Location 的价值

> `Location` 头实现了 REST 的 HATEOAS 原则——客户端不需要自己拼 URL，服务器直接告诉你新资源在哪里。这对自动化客户端（SDK、测试框架）尤其有价值。

### 307/308 + Location 示例

```JavaScript
HTTP/1.1 308 Permanent Redirect
Location: https://api.example.com/v2/users
```

> [!important] 重定向中 Location 是必须的

> 没有 Location 头的 3xx 重定向是无效的——客户端不知道该去哪里。

## 2. Retry-After 头

### 适用状态码

|状态码|Retry-After 含义|推荐度|
|---|---|---|
|429 Too Many Requests|多久后可以再次请求|**强烈推荐**|
|503 Service Unavailable|服务预计何时恢复|**强烈推荐**|
|301 Moved Permanently|迁移计划时间|少见|

### 两种格式

```JavaScript
# 格式 1：秒数（推荐，简单明确）
Retry-After: 30

# 格式 2：HTTP-date（适合计划维护）
Retry-After: Wed, 21 Oct 2026 07:28:00 GMT
```

### 429 + Retry-After 完整示例

```Python
from fastapi import Request
from fastapi.responses import JSONResponse
import time

# 简单的令牌桶限流
class RateLimiter:
    def __init__(self, max_requests: int, window_seconds: int):
        self.max_requests = max_requests
        self.window = window_seconds
        self.requests = {}  # ip -> [timestamps]
    
    def check(self, client_ip: str) -> tuple[bool, int]:
        now = time.time()
        reqs = self.requests.get(client_ip, [])
        reqs = [t for t in reqs if now - t < self.window]
        if len(reqs) >= self.max_requests:
            retry_after = int(self.window - (now - reqs[0]))
            return False, retry_after
        reqs.append(now)
        self.requests[client_ip] = reqs
        return True, 0

limiter = RateLimiter(max_requests=60, window_seconds=60)

@app.middleware("http")
async def rate_limit_middleware(request: Request, call_next):
    client_ip = request.client.host
    allowed, retry_after = limiter.check(client_ip)
    if not allowed:
        return JSONResponse(
            status_code=429,
            content={
                "code": "RATE_LIMIT_EXCEEDED",
                "message": f"请求过于频繁，请 {retry_after} 秒后重试"
            },
            headers={"Retry-After": str(retry_after)}
        )
    return await call_next(request)
```

> [!faq] 为什么 Retry-After 重要？

> 没有 `Retry-After` 的 429 让客户端**不知道该等多久**。好的客户端 SDK 会读取 `Retry-After` 头自动实现退避等待；没有这个头，客户端只能自己猜测（通常导致不必要的重试风暴）。

## 3. WWW-Authenticate 头

### 适用状态码

|状态码|WWW-Authenticate 作用|是否必须|
|---|---|---|
|401 Unauthorized|告知客户端需要什么认证方案|**RFC 要求必须**|
|407 Proxy Authentication Required|代理认证|必须（代理场景）|

### RFC 9110 §15.5.2

"A server generating a 401 response MUST send a WWW-Authenticate header field containing at least one challenge applicable to the target resource."

### 常见方案示例

```JavaScript
# Bearer Token（OAuth 2.0 / JWT）
WWW-Authenticate: Bearer realm="api"
WWW-Authenticate: Bearer realm="api", error="invalid_token", error_description="The access token expired"

# Basic Auth
WWW-Authenticate: Basic realm="admin-panel"

# 多方案（客户端可选择）
WWW-Authenticate: Bearer realm="api"
WWW-Authenticate: Basic realm="api"
```

### FastAPI 实现

```Python
from fastapi import Depends, HTTPException
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

security = HTTPBearer(auto_error=False)

async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security)
):
    if credentials is None:
        raise HTTPException(
            status_code=401,
            detail={"code": "AUTH_REQUIRED", "message": "需要认证"},
            headers={"WWW-Authenticate": 'Bearer realm="api"'}
        )
    
    token = credentials.credentials
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
    except jwt.ExpiredSignatureError:
        raise HTTPException(
            status_code=401,
            detail={"code": "TOKEN_EXPIRED", "message": "令牌已过期"},
            headers={"WWW-Authenticate": 'Bearer error="invalid_token", error_description="The access token expired"'}
        )
    except jwt.InvalidTokenError:
        raise HTTPException(
            status_code=401,
            detail={"code": "INVALID_TOKEN", "message": "无效的令牌"},
            headers={"WWW-Authenticate": 'Bearer error="invalid_token"'}
        )
    
    return payload
```

> [!important] 思辨：为什么很多项目忽略 WWW-Authenticate？

> 因为大多数 SPA/移动端不依赖浏览器的原生认证弹窗，开发者觉得这个头"没用"。但它对**自动化客户端**（SDK、CLI 工具、OAuth 库）非常重要——这些工具读取 `WWW-Authenticate` 来自动决定是刷新 token 还是重新登录。忽略它会让 OAuth 2.0 的自动刷新流程失效。

## 检查清单

- [ ] 201 响应是否包含 `Location` 头？

- [ ] 3xx 重定向是否包含 `Location` 头？

- [ ] 429 响应是否包含 `Retry-After` 头？

- [ ] 503 响应是否包含 `Retry-After` 头？

- [ ] 401 响应是否包含 `WWW-Authenticate` 头？

- [ ] 304 响应是否包含缓存相关头（`ETag`/`Cache-Control`）？

- [ ] 405 响应是否包含 `Allow` 头？
