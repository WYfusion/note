## 概述

本页完整枚举 HTTP 状态码五大分类的所有标准码，并阐释 RFC 9110 统一修订的背景与设计哲学。

> [!info] RFC 9110 背景

> 2022 年 6 月，IETF 发布 RFC 9110（HTTP Semantics），统一取代了 RFC 7231、7232、7233、7235 等多个旧规范。最重要的变化之一：将所有 HTTP 语义（方法、状态码、头部）合并到一个文档中，消除了分散在多个 RFC 之间的歧义与不一致。对状态码而言，RFC 9110 未新增码值，但**修正了部分描述**（如 422 从 "Unprocessable Entity" 改为 "Unprocessable Content"）并将原属 WebDAV 的 421、422 正式纳入核心标准。

---

## 1.1 信息类 1xx（少用）

|码|名称|用途|
|---|---|---|
|100|Continue|客户端发送大 body 前的探测；服务端确认可以继续|
|101|Switching Protocols|协议升级（如 HTTP → WebSocket）|

> [!tip] 思辨：REST API 为何几乎不主动设计 1xx？

> 1xx 是**中间响应**，不是最终响应。REST API 的交互模型是"请求→最终响应"，中间态由 HTTP 库和服务器自动处理。开发者无需也不应显式返回 100/101——框架和网关已经在协议层自动完成了。

---

## 1.2 成功类 2xx（最常用）

|码|名称|典型场景|
|---|---|---|
|200|OK|查询成功、普通更新成功、返回内容的通用成功|
|201|Created|新资源已创建，应配合 `Location` 头返回新资源 URI|
|202|Accepted|已接收但尚未完成处理（异步任务）|
|204|No Content|成功但禁止携带响应体（DELETE 成功、空更新）|
|206|Partial Content|分块传输 / 断点续传，配合 `Content-Range`|

> [!faq] 为什么不统一用 200？

> 200 是通用成功码，但**语义模糊**。201 明确告知"资源刚被创建"，202 告知"异步执行中"，204 告知"成功但别解析 body"——这些语义差异直接影响客户端的后续行为（是否轮询、是否解析响应、是否跳转到新资源）。

---

## 1.3 重定向类 3xx（Web/API 均常见）

|码|名称|方法保留|典型场景|
|---|---|---|---|
|301|Moved Permanently|❌ 不保证|永久迁移（SEO 常用）|
|302|Found|❌ 不保证|临时跳转；历史兼容行为复杂|
|303|See Other|强制 GET|POST 后跳转到结果页|
|304|Not Modified|—|缓存协商命中，不返回实体|
|307|Temporary Redirect|✅ 保留|临时跳转且保持原方法/body|
|308|Permanent Redirect|✅ 保留|永久跳转且保持原方法/body|

> [!important] 方法保留问题——3xx 设计的历史包袱

> RFC 2616 时代的 301/302 没有严格规定重定向后是否保留原方法，导致不同浏览器行为不一。RFC 9110 通过 307/308 **明确**了方法保留语义，解决了 POST/PUT 等非 GET 方法重定向的歧义。API 中涉及非 GET 方法跳转时，**必须优先用 307/308**。

---

## 1.4 客户端错误类 4xx（最关键）

**高频核心码**：

|码|名称|一句话语义|
|---|---|---|
|400|Bad Request|请求格式/报文层就有问题|
|401|Unauthorized|未认证 / 凭证无效|
|403|Forbidden|已识别身份但无权限|
|404|Not Found|资源不存在|
|405|Method Not Allowed|资源存在，但方法不允许|
|409|Conflict|资源状态冲突|
|415|Unsupported Media Type|Content-Type 不支持|
|422|Unprocessable Content|语法正确但语义校验不通过|
|429|Too Many Requests|限流|

**补充码**：

|码|名称|一句话语义|
|---|---|---|
|406|Not Acceptable|Accept 头协商失败|
|408|Request Timeout|服务端等客户端超时|
|410|Gone|明确永久消失|
|412|Precondition Failed|条件请求头不满足|
|413|Content Too Large|请求体过大|
|414|URI Too Long|URI 过长|
|416|Range Not Satisfiable|Range 请求范围无效|
|421|Misdirected Request|请求打到错误的连接上下文|
|423|Locked|WebDAV 资源锁定|
|424|Failed Dependency|WebDAV 依赖操作失败|
|425|Too Early|防 TLS 重放攻击|
|426|Upgrade Required|需协议升级|
|428|Precondition Required|必须带条件请求头|
|431|Request Header Fields Too Large|请求头过大|
|451|Unavailable For Legal Reasons|因法律原因不可用|

> [!tip] 422 的演进

> 422 原属 WebDAV（RFC 4918），名称为 "Unprocessable Entity"。RFC 9110 将其正式纳入核心标准并更名为 "Unprocessable Content"，语义更准确——强调的是**内容**不可处理，不限于 "实体"。

## 1.5 服务端错误类 5xx

|码|名称|一句话语义|
|---|---|---|
|500|Internal Server Error|服务自身代码/逻辑炸了|
|501|Not Implemented|不支持该功能/方法（≠业务未开发）|
|502|Bad Gateway|网关收到上游无效响应|
|503|Service Unavailable|临时不可用（维护/过载/熔断）|
|504|Gateway Timeout|网关等待上游超时|
|505|HTTP Version Not Supported|协议版本不支持|
|507|Insufficient Storage|WebDAV 存储不足|
|508|Loop Detected|WebDAV 循环检测|
|511|Network Authentication Required|网络层接入认证（如 WiFi Portal）|

> [!faq] 500 和 502 的本质区别？

> **500 = 你自己的代码出了未处理异常**；**502 = 你的代码没问题，但你调用的上游服务返回了一个无法解析的响应**。在微服务架构中，这个区分至关重要——502 意味着问题在**上游**，排障方向完全不同。
