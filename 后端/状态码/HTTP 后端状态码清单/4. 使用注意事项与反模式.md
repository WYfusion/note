## 概述

9 条实战军规 + HTTP 状态码与业务错误码的双层设计方法论。知道"用什么码"只是第一步，知道"不要怎么用"同样关键。

> [!important] 思辨：为什么需要"反模式"清单？

> 软件工程中，正确的做法往往有多种，但错误的做法却高度相似且反复出现。明确列出反模式，比正面说"应该怎样"更有效——因为反模式通常有**即时的便利性**（比如全返 200 省事），需要靠认知来抵抗惯性。

---

## 4.1 不要把业务失败全部塞成 200

**反模式**：

```Python
# ❌ 所有请求都返回 200，错误藏在 body 里
return JSONResponse(
    status_code=200,
    content={"code": 1001, "msg": "用户不存在"}
)
```

**问题**：

- Nginx / CDN 的错误率统计全部失效

- Prometheus 的 HTTP 状态码指标变成摆设

- Sentry / 告警系统无法基于状态码触发

- 客户端 SDK（如 axios）的 `catch` 分支永远不会触发

**正确做法**：

```Python
# ✅ 用正确的 HTTP 状态码 + 业务细节放 body
raise HTTPException(
    status_code=404,
    detail={"code": "USER_NOT_FOUND", "message": "用户不存在"}
)
```

---

## 4.2 不要把所有客户端问题都塞成 400

|错误场景|❌ 错误|✅ 正确|
|---|---|---|
|参数语义不合法|400|**422**|
|未认证|400|**401**|
|无权限|400|**403**|
|资源不存在|400|**404**|
|状态冲突|400|**409**|

> [!faq] 为什么要细分？

> 400 的语义是"请求本身在协议/格式层面有问题"。如果 token 过期也返回 400，客户端无法区分"该重新登录"还是"该修正请求参数"。细分状态码 = 让客户端**自动化处理**不同类型的错误成为可能。

---

## 4.3 不要把所有服务端问题都塞成 500

|错误场景|❌ 错误|✅ 正确|
|---|---|---|
|上游返回坏响应|500|**502**|
|过载 / 维护|500|**503**|
|上游超时|500|**504**|

> [!tip] 运维收益

> 502/503/504 的区分让运维团队在告警时能**立即判断排障方向**：502 找上游、503 扩容、504 优化超时。全塞成 500 则需要逐条查日志。

---

## 4.4 401 要配合认证机制

返回 401 时应同时设置 `WWW-Authenticate` 响应头，告知客户端使用什么认证方案：

```JavaScript
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api", error="invalid_token"
```

> [!info] RFC 9110 §15.5.2

> "A server generating a 401 response MUST send a WWW-Authenticate header field containing at least one challenge applicable to the target resource." 虽然很多实现忽略了这一点，但严格遵守可以让 OAuth 客户端库自动处理 token 刷新。

---

## 4.5 204 / 304 不能乱带响应体

- **204**：RFC 9110 明确规定"A server MUST NOT send content in a response to a request with a 204 status code"。违反这一点可能导致某些 HTTP 客户端解析异常

- **304**：同理，304 不应包含 body，但**必须包含**与 200 相同的缓存相关头（`ETag`、`Cache-Control` 等）

---

## 4.6 201 最好配资源定位信息

```JavaScript
HTTP/1.1 201 Created
Location: /api/users/12345
Content-Type: application/json

{"id": 12345, "name": "sion"}
```

> [!tip] 为什么 Location 头重要？

> 客户端创建资源后最常见的后续操作是"查看/编辑刚创建的资源"。`Location` 头让客户端无需解析 body 就能拿到新资源的标准 URI，实现了**HATEOAS（超媒体即应用状态引擎）**的基本要求。

---

## 4.7 429 / 503 最好给重试信息

```JavaScript
HTTP/1.1 429 Too Many Requests
Retry-After: 30

{"code": "RATE_LIMIT", "message": "请求过于频繁，请 30 秒后重试"}
```

```JavaScript
HTTP/1.1 503 Service Unavailable
Retry-After: Wed, 21 Oct 2026 07:28:00 GMT

{"code": "MAINTENANCE", "message": "系统维护中"}
```

> [!important] 429 vs 503 的区分

> - **429** = 你个人/客户端请求太多（限流）→ `Retry-After` 告诉你多久后可以再试

> - **503** = 服务整体不可用（维护/过载） → `Retry-After` 告诉所有人何时恢复

> 区分它们让客户端知道是"我被限了"还是"服务挂了"，决定是退避重试还是直接报错给用户。

---

## 4.8 状态码要稳定、统一、可预测

**原则**：

- 同一类失败在全项目保持一致（如所有认证失败都是 401，不要有的路由返 401 有的返 403）

- 同一资源族的 CRUD 规则保持一致（如所有集合资源的 POST 创建都返 201）

- 在 OpenAPI / Swagger 文档中显式声明所有可能的状态码

> [!tip] 思辨：稳定性 > 精确性

> 如果团队对某个边界有争议（比如该用 400 还是 422），**达成一致并坚持**比"每次都选最精确的码"更重要。前后端对齐的成本远高于"码不够精确"的成本。

---

## 4.9 区分 HTTP 状态码与业务错误码

**双层设计**：

```JavaScript
┌─────────────────────────────────────────┐
│         HTTP 层（状态码）                 │
│   协议级分类：成功/客户端错误/服务端错误    │
├─────────────────────────────────────────┤
│         业务层（错误码 + 消息）            │
│   领域级细分：具体是什么业务规则失败       │
└─────────────────────────────────────────┘
```

**典型组合**：

|HTTP 状态码|业务错误码|含义|
|---|---|---|
|409|`USER_ALREADY_EXISTS`|用户已存在|
|422|`INVALID_EMAIL_FORMAT`|邮箱格式无效|
|403|`PLAN_NOT_ALLOWED`|套餐不支持此功能|
|429|`RATE_LIMIT_EXCEEDED`|请求频率超限|
|502|`PAYMENT_GATEWAY_ERROR`|支付网关异常|

> [!important] 思辨：为什么需要双层而非只用业务码？

> HTTP 状态码是**所有基础设施**（网关、CDN、监控、负载均衡器）共同理解的语言。业务码只有你的应用代码和客户端 SDK 理解。双层设计让基础设施能做粗粒度分类（成功/失败/限流/熔断），应用层做细粒度分类（具体什么业务规则）。
