## 1xx 信息类

1xx 是 HTTP 的**中间响应码**，表示服务器已收到请求的初始部分，客户端可以继续发送。

|码|名称|RFC 9110 定义|实际使用|
|---|---|---|---|
|100|Continue|服务端已收到请求头，客户端可以继续发 body|大文件上传时由 HTTP 库自动处理|
|101|Switching Protocols|响应客户端的 `Upgrade` 请求|WebSocket 握手时自动触发|
|103|Early Hints|提前发送 `Link` 头让浏览器预加载资源|较新，主要用于浏览器性能优化|

> [!tip] 开发者基本不需要手动处理 1xx

> 1xx 码由 HTTP 协议栈（框架/库/网关）自动管理。在 REST API 设计中，你不应该也不需要主动返回 1xx。唯一的例外是如果你在实现底层 HTTP 服务器或代理。

## 2xx 成功类

### 200 OK

**语义**：请求已成功，响应体包含请求结果。

**RFC 9110 §15.3.1**："The 200 (OK) status code indicates that the request has succeeded. The content sent in a 200 response depends on the request method."

**使用场景**：

- GET 查询成功

- PUT/PATCH 更新成功且返回更新后的资源

- 任何需要返回内容的通用成功

> [!faq] 200 的"坑"

> 200 是最被滥用的状态码。很多团队把所有响应都返回 200，把错误信息藏在 body 里。这破坏了 HTTP 协议的语义契约，让所有依赖状态码的基础设施（网关、CDN、监控、Sentry）全部失效。

---

### 201 Created

**语义**：请求成功且**新资源已被创建**。

**RFC 9110 §15.3.2**："The primary resource created by the request is identified by either a Location header field in the response or, if no Location header field is received, by the target URI."

**最佳实践**：

```Python
# FastAPI 示例
@app.post("/users", status_code=201)
async def create_user(user: UserCreate):
    new_user = await user_service.create(user)
    return JSONResponse(
        status_code=201,
        content=new_user.dict(),
        headers={"Location": f"/api/users/{new_user.id}"}
    )
```

> [!important] 201 必须配合资源定位

> RFC 9110 要求 201 响应应通过 `Location` 头或目标 URI 标识新创建的主资源。这不仅是协议要求，也是实用需求——客户端创建资源后的第一件事通常是"查看/编辑刚创建的东西"。

---

### 202 Accepted

**语义**：请求已被接受进行处理，但处理**尚未完成**。

**关键特征**：

- 不承诺最终结果（可能成功也可能失败）

- 适用于异步任务、消息队列、批处理

- 响应体中应包含任务 ID 或状态查询 URL

```Python
@app.post("/exports", status_code=202)
async def start_export(config: ExportConfig):
    task_id = await export_service.enqueue(config)
    return {
        "task_id": task_id,
        "status_url": f"/api/exports/{task_id}/status"
    }
```

> [!tip] 202 的轮询模式

> 返回 202 后，客户端通常需要轮询 `status_url` 来获取最终结果。更现代的做法是配合 WebSocket 或 Webhook 进行回调通知，避免轮询开销。

---

### 204 No Content

**语义**：请求成功但**服务器不返回任何内容**。

**RFC 9110 §15.3.5**："A server MUST NOT send content in a response to a request with a 204 status code."

**典型场景**：

- DELETE 成功

- PUT/PATCH 更新成功但不需要返回更新后的资源

- 批量操作成功

> [!important] 204 的严格约束

> 204 **禁止**携带响应体。如果你的框架/代码不小心在 204 响应中放了 body，某些 HTTP 客户端（特别是严格实现的）会解析异常。Python 的 `httpx` 和 Go 的 `net/http` 都会正确忽略 204 的 body，但不是所有实现都如此。

---

### 206 Partial Content

**语义**：服务器只返回了资源的一部分（响应 `Range` 请求）。

**使用场景**：

- 大文件下载的断点续传

- 视频/音频流的分段请求

- 分页下载（较少见）

**必须配合的头**：

- `Content-Range: bytes 0-999/5000`

- `Accept-Ranges: bytes`

> [!info] 206 在 TTS/语音工程中的价值

> 在语音合成或音频流服务中，206 是实现流式音频传输的基础。客户端可以请求音频文件的特定片段（如 `Range: bytes=1024000-2048000`），实现"边下边播"而不需要等完整文件下载完成。
