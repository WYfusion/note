## 429 的语义

RFC 6585 §4: "The 429 status code indicates that the user has sent too many requests in a given amount of time ('rate limiting')."

429 表示**客户端**请求频率过高。这是一个**暂时性**状态——降低请求频率后就能正常访问。

## 限流算法对比

### 1. 固定窗口计数器

**原理**：将时间划分为固定窗口（如每分钟），在窗口内计数。

```Python
import time
from collections import defaultdict

class FixedWindowLimiter:
    def __init__(self, max_requests: int, window_seconds: int):
        self.max_requests = max_requests
        self.window = window_seconds
        self.counters: dict[str, tuple[int, int]] = {}  # key -> (count, window_start)
    
    def allow(self, key: str) -> tuple[bool, int]:
        now = int(time.time())
        window_start = now - (now % self.window)
        
        count, ws = self.counters.get(key, (0, window_start))
        if ws != window_start:
            count, ws = 0, window_start  # 新窗口
        
        if count >= self.max_requests:
            retry_after = window_start + self.window - now
            return False, retry_after
        
        self.counters[key] = (count + 1, ws)
        return True, 0
```

> [!faq] 固定窗口的边界问题

> 如果用户在窗口末尾发送 N 个请求，然后在下一个窗口开头又发送 N 个请求，实际上在 2 秒内发送了 2N 个请求。这就是"窗口边界突刺"问题。

### 2. 滑动窗口计数器

**原理**：使用加权方式平滑窗口边界。

```Python
class SlidingWindowLimiter:
    def __init__(self, max_requests: int, window_seconds: int):
        self.max_requests = max_requests
        self.window = window_seconds
        self.prev_counts: dict[str, int] = defaultdict(int)
        self.curr_counts: dict[str, int] = defaultdict(int)
        self.window_starts: dict[str, int] = {}
    
    def allow(self, key: str) -> tuple[bool, int]:
        now = int(time.time())
        window_start = now - (now % self.window)
        
        if self.window_starts.get(key) != window_start:
            self.prev_counts[key] = self.curr_counts.get(key, 0)
            self.curr_counts[key] = 0
            self.window_starts[key] = window_start
        
        # 加权计算：当前窗口已过比例
        elapsed_ratio = (now - window_start) / self.window
        estimated = self.prev_counts[key] * (1 - elapsed_ratio) + self.curr_counts[key]
        
        if estimated >= self.max_requests:
            retry_after = int(self.window * (1 - elapsed_ratio)) + 1
            return False, retry_after
        
        self.curr_counts[key] += 1
        return True, 0
```

### 3. 令牌桶（Token Bucket）

**原理**：以固定速率向桶中添加令牌，每次请求消耗一个令牌。桶有最大容量（允许突发）。

```Python
import time

class TokenBucketLimiter:
    def __init__(self, rate: float, capacity: int):
        """
        rate: 每秒补充的令牌数
        capacity: 桶的最大容量（允许的突发量）
        """
        self.rate = rate
        self.capacity = capacity
        self.buckets: dict[str, tuple[float, float]] = {}  # key -> (tokens, last_time)
    
    def allow(self, key: str) -> tuple[bool, float]:
        now = time.time()
        tokens, last_time = self.buckets.get(key, (self.capacity, now))
        
        # 补充令牌
        elapsed = now - last_time
        tokens = min(self.capacity, tokens + elapsed * self.rate)
        
        if tokens < 1:
            # 计算需要等待多久才能获得一个令牌
            retry_after = (1 - tokens) / self.rate
            return False, retry_after
        
        self.buckets[key] = (tokens - 1, now)
        return True, 0
```

> [!tip] 令牌桶 vs 滑动窗口

> - **令牌桶**允许**突发流量**——桶满时可以一次性消耗多个令牌。适合 API 场景（允许短期突发但限制平均速率）。

> - **滑动窗口**更**平滑**——严格限制任何窗口内的请求数。适合需要严格限速的场景。

> - **实际选择**：大多数 API 限流用令牌桶。Nginx、Redis 限流模块、云厂商 API Gateway 默认都是令牌桶或漏桶变体。

## 算法对比

|**算法**|**窗口边界突刺**|**允许突发**|**内存**|**实现复杂度**|
|---|---|---|---|---|
|固定窗口|❌ 有|❌ 不允许|低|极简|
|滑动窗口|✅ 平滑|❌ 不允许|低|简单|
|令牌桶|✅ 平滑|✅ 允许|低|中等|
|漏桶|✅ 平滑|❌ 严格|可能需要队列|中等|

## Retry-After 的正确使用

### RFC 6585 的要求

> "The response representations SHOULD include details explaining the condition, and MAY include a Retry-After header indicating how long to wait before making a new request."

虽然 RFC 用的是 MAY（可选），但在实践中 **Retry-After 是必须的**——没有它，客户端只能盲猜重试间隔。

### 响应头设计

```Python
# 完整的 429 响应
from fastapi.responses import JSONResponse

def rate_limit_response(retry_after: int, limit: int, remaining: int, reset: int) -> JSONResponse:
    return JSONResponse(
        status_code=429,
        content={
            "code": "RATE_LIMIT_EXCEEDED",
            "message": f"请求过于频繁，请 {retry_after} 秒后重试",
            "detail": f"限额：每分钟 {limit} 次"
        },
        headers={
            "Retry-After": str(retry_after),
            # 以下为非标准但广泛使用的限流头
            "X-RateLimit-Limit": str(limit),       # 限额
            "X-RateLimit-Remaining": str(remaining), # 剩余
            "X-RateLimit-Reset": str(reset),         # 重置时间戳
        }
    )
```

> [!important] X-RateLimit 头族

> `X-RateLimit-*` 不是 HTTP 标准，但已成为**事实标准**。GitHub、Twitter/X、Stripe 等主流 API 都使用这些头。它们让客户端能在**被限流之前**就知道自己的配额状态，主动调节请求频率。

## 429 vs 503 的区别

|维度|429|503|
|---|---|---|
|针对谁|**特定客户端**（你请求太多了）|**所有人**（整个服务不可用）|
|其他客户端|不受影响|都受影响|
|原因|客户端请求频率过高|服务过载/维护/依赖故障|
|责任归属|4xx = 客户端问题|5xx = 服务端问题|
|Retry-After|通常较短（秒级）|可能较长（分钟级）|

> [!important] 思辨：限流用 429 还是 503？

> 如果是**针对特定客户端的限流**（API Key 级别、IP 级别） → **429**。

> 如果是**全局过载保护**（服务端即将崩溃，对所有请求都限流） → **503**。

> 某些团队在全局过载时也用 429，这是不准确的。503 传达的信号是"我整个人不行了"，429 传达的是"就你请求太多了"。
