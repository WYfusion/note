## 概述

4xx 是**最丰富、最关键**的状态码类别。它表示"请求有问题，客户端需要修改后重试"。正确使用 4xx 的核心在于**精确归类客户端的错误类型**，让客户端能自动化处理不同场景。

---

## 核心九码详解

### 400 Bad Request

**语义**：请求在**协议/格式/结构层面**就有问题，服务器无法理解。

**典型触发**：

- JSON 语法错误（缺逗号、未闭合引号）

- 缺少必要的请求头

- URL 编码错误

- 请求结构不符合协议规范

> [!faq] 400 不是万能垃圾桶

> 很多团队把所有客户端错误都返回 400。这等于放弃了 HTTP 协议提供的精细分类能力。客户端收到 400 只知道"我的请求有问题"，但不知道是格式问题、认证问题、权限问题还是参数问题。

---

### 401 Unauthorized

**语义**：请求缺少有效的认证凭证。

**RFC 9110 要求**：401 响应**必须**包含 `WWW-Authenticate` 头。

```JavaScript
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api", error="invalid_token", error_description="The access token expired"
```

**典型触发**：没带 token、token 过期、token 签名无效、API Key 错误。

---

### 403 Forbidden

**语义**：服务器理解请求且**已识别身份**，但拒绝执行。

**与 401 的关键区别**：401 = 身份未确认；403 = 身份已确认但无权。

**安全考量**：某些场景下可以用 404 替代 403 来隐藏资源存在性（如 GitHub 的私有仓库策略）。

---

### 404 Not Found

**语义**：目标资源不存在。

**注意**：404 **不承诺**资源永远不存在（将来可能会有）。如果需要表达"永久消失"，用 410。

---

### 409 Conflict

**语义**：请求与资源的**当前状态**冲突。

**典型场景**：

- 重复创建（唯一约束违反）

- 乐观锁版本冲突

- 状态机不允许的状态转换（如已取消的订单不能支付）

```Python
# FastAPI 乐观锁冲突示例
@app.put("/articles/{id}")
async def update_article(id: int, data: ArticleUpdate):
    article = await repo.get(id)
    if article.version != data.expected_version:
        raise HTTPException(
            status_code=409,
            detail={
                "code": "VERSION_CONFLICT",
                "message": "资源已被其他人修改",
                "current_version": article.version
            }
        )
```

---

### 415 Unsupported Media Type

**语义**：请求的 `Content-Type` 不被接口支持。

**典型触发**：接口只接受 `application/json`，客户端发了 `application/xml` 或 `text/plain`。

---

### 422 Unprocessable Content

**语义**：请求格式正确、可解析，但**内容在语义/业务层面不合法**。

**RFC 9110 更名**：从 "Unprocessable Entity"（WebDAV 时代）改为 "Unprocessable Content"。

**典型触发**：

- 邮箱格式无效

- 数字超出范围

- 枚举值不在允许列表中

- 日期逻辑错误（结束日期早于开始日期）

> [!important] FastAPI 与 422

> FastAPI 的 Pydantic 校验自动返回 422。这是正确的——Pydantic 校验的是**语义合法性**（字段类型、格式、范围），而非报文语法。

---

### 429 Too Many Requests

**语义**：客户端在给定时间内发送了过多请求。

**最佳实践**：

- 必须配合 `Retry-After` 头

- 应在响应体中说明限流策略（如"每分钟最多 60 次"）

- 区分"用户级限流"和"IP 级限流"

---

### 405 Method Not Allowed

**语义**：请求的 HTTP 方法对目标资源不被允许。

**要求**：405 响应**必须**包含 `Allow` 头，列出允许的方法。

```JavaScript
HTTP/1.1 405 Method Not Allowed
Allow: GET, POST, HEAD
```

---

## 补充码速览

|码|关键语义|典型场景|
|---|---|---|
|406|Accept 协商失败|客户端要 XML 但服务只提供 JSON|
|408|服务端等客户端超时|慢速客户端长时间不发数据|
|410|永久消失|API 版本下线、明确废弃的资源|
|412|前置条件失败|`If-Match` ETag 不匹配|
|413|请求体过大|上传文件超限|
|428|必须条件请求|服务端强制要求 `If-Match`|
|431|请求头过大|Cookie 累积过多|
|451|法律限制|GDPR、版权审查、地域封锁|
