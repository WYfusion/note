## 核心问题

HTTP 重定向家族的混乱源于**早期规范与浏览器实现的不一致**。理解这段历史才能理解为什么会有 5 个重定向码。

---

## 时间线

### 1996: HTTP/1.0 (RFC 1945)

- 引入 **301** Moved Permanently 和 **302** Moved Temporarily

- 规范对 302 的描述含糊："If the 302 status code is received in response to a request other than GET or HEAD, the user agent MUST NOT automatically redirect the request unless it can be confirmed by the user."

- **现实**：几乎所有浏览器都无视了这条规定，将 302 响应中的 POST 改为 GET 后重定向

### 1999: HTTP/1.1 (RFC 2616)

- 承认了 302 的实现混乱，试图通过引入新码来修复

- 引入 **303** See Other："明确要求客户端用 GET 访问重定向目标"

- 引入 **307** Temporary Redirect："必须保留原始方法和请求体"

- 但 **301** 和 **302** 的方法变更行为仍然未被严格定义

> [!important] RFC 2616 的核心贡献

> RFC 2616 的策略不是修复 301/302，而是**绕过它们**——创造新码来填补语义空缺。这是一个务实但不优雅的决定。它承认了"规范无法覆盖已有的实现"这个残酷现实。

### 2014: RFC 7231 (HTTP/1.1 Semantics)

- HTTP/1.1 规范重构，将语义从传输层分离

- 明确承认 301/302 对非安全方法的行为"在实践中已经改变"：浏览器会将 POST 变为 GET

- 对 303/307 的语义进行了更精确的定义

### 2015: RFC 7538

- 引入 **308** Permanent Redirect

- 308 与 301 的关系 = 307 与 302 的关系

- 填补了"永久 + 保留方法"这个空缺

### 2022: RFC 9110 (HTTP Semantics)

- 统一了所有 HTTP 语义定义

- 正式化了五个重定向码的完整语义矩阵

- 对 302 的描述："For historical reasons, a user agent MAY change the request method from POST to GET for the subsequent request."

---

## 语义矩阵的最终形态

||**临时**|**永久**|
|---|---|---|
|**可能改方法**（历史行为）|302 Found (1996)|301 Moved Permanently (1996)|
|**强制 GET**（明确语义）|303 See Other (1999)|—|
|**保留方法**（明确语义）|307 Temporary Redirect (1999)|308 Permanent Redirect (2015)|

> [!faq] 为什么没有"永久 + 强制 GET"的码？

> 因为在实践中 301 已经扮演了这个角色——浏览器对 301 的实际行为就是将 POST 改为 GET。所以不需要一个新码。303 只需要临时版本，因为"永久强制 GET"的场景已经被 301 的实际行为覆盖了。

## 对 API 开发的影响

### 为什么 301/302 对 API 不安全

```Python
import httpx

# 危险场景：POST 请求被 301 重定向
# 某些 HTTP 客户端会将 POST 变为 GET，丢失请求体
response = httpx.post(
    "https://old-api.example.com/users",
    json={"name": "test"},
    follow_redirects=True  # 如果服务器返回 301
)
# 实际发送：GET https://new-api.example.com/users（body 丢失！）
```

### httpx 的行为

httpx（Python 现代 HTTP 客户端）对重定向的处理：

|状态码|原始方法|重定向后方法|
|---|---|---|
|301|POST|**GET**（丢失 body）|
|302|POST|**GET**（丢失 body）|
|303|POST|**GET**（预期行为）|
|307|POST|**POST**（保留 body）|
|308|POST|**POST**（保留 body）|

> [!important] API 迁移的正确做法

> 当你需要迁移 API 端点时：

> - 如果只有 GET 请求 → 301 没问题

> - 如果涉及 POST/PUT/PATCH/DELETE → **必须用 307/308**

> - 如果是 POST 表单提交后查看结果 → 303

## RFC 9110 的关键引用

### §15.4.2 (301)

> "Note: For historical reasons, a user agent MAY change the request method from POST to GET for the subsequent request. If this behavior is undesired, the 308 (Permanent Redirect) status code can be used instead."

### §15.4.4 (303)

> "This status code is applicable to any HTTP method. It is primarily used to allow the output of a POST action to redirect the user agent to a selected resource."

### §15.4.8 (307)

> "The user agent MUST NOT change the request method if it performs an automatic redirection to that URI."

### §15.4.9 (308)

> "This status code is similar to 301 (Moved Permanently), except that it does not allow changing the request method from POST to GET."

> [!tip] 总结

> HTTP 重定向的历史是一部**规范追赶实现**的历史。RFC 9110 最终承认了现实，用 5 个码覆盖了 2×2 矩阵（临时/永久 × 改方法/保留方法）+ 1 个特殊用途（303 强制 GET）。理解这段历史，就不会在 API 迁移中误用 301/302。
