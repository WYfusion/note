## 概述

按实际开发场景将状态码分为四大速查集，覆盖 CRUD、Web 缓存、并发控制、以及较新/补充码。

> [!tip] 思辨：为什么要按"场景"而非"码值"组织？

> 开发中的决策起点永远是"我现在要干什么"，而不是"我背了哪些码"。按场景组织能让你在面对具体问题（创建资源？重定向？限流？）时快速定位到正确的状态码子集，而不是在 60 多个码里线性扫描。

---

## 2.1 CRUD / REST API 最常用主集

|**操作**|**成功**|**常见失败**|
|---|---|---|
|GET 查询|200|400 / 401 / 403 / 404|
|POST 创建|201|400 / 409 / 415 / 422|
|PUT/PATCH 更新|200 / 204|400 / 404 / 409 / 412 / 422|
|DELETE 删除|204 (200)|401 / 403 / 404 / 409|
|异步任务|202|400 / 500|

**通用失败码**（所有操作均可能触发）：

- `401` — 未认证

- `403` — 无权限

- `429` — 限流

- `500` — 服务内部异常

- `502` / `503` / `504` — 上游/基础设施问题

---

## 2.2 Web / 缓存 / 重定向常用主集

|场景|状态码|关键行为|
|---|---|---|
|永久迁移（SEO）|301|搜索引擎更新索引|
|临时跳转（兼容）|302|浏览器可能改变方法|
|POST 后查看结果|303|强制 GET|
|缓存未修改|304|不返回 body|
|临时跳转（保留方法）|307|保留 POST/PUT body|
|永久跳转（保留方法）|308|保留 POST/PUT body|

> [!important] 实践建议

> - 纯浏览器场景可以继续用 301/302

> - API 场景涉及非 GET 方法跳转时 → 只用 307/308

> - 缓存协商 304 需要配合 `ETag` 或 `Last-Modified`

## 2.3 并发控制 / 幂等 / 条件请求常用

|场景|状态码|语义|
|---|---|---|
|乐观锁冲突|409|资源当前状态与请求意图冲突|
|带了条件但不满足|412|`If-Match` / `If-Unmodified-Since` 失败|
|服务器强制条件请求|428|必须带 `If-Match` 等头|
|请求频率过高|429|配合 `Retry-After` 头|
|缓存命中|304|`If-None-Match` / `If-Modified-Since` 成功|
|分块传输 / 断点续传|206|`Range` • `Content-Range`|

> [!faq] 409 vs 412 vs 428 怎么选？

> **三角口诀**：缺条件 = 428 → 条件没过 = 412 → 一般性状态冲突 = 409。这三个码覆盖了从"客户端没带条件"到"条件不满足"到"资源本身冲突"的完整链路。

## 2.4 较新 / 补充但应认识

|码|场景|说明|
|---|---|---|
|421|HTTP/2 连接复用错误|请求打到了错误的服务端连接上下文|
|425|TLS 0-RTT 重放防护|服务器不愿冒请求重放风险|
|431|超大请求头|Cookie 过多、认证头过大等|
|451|法律合规|GDPR、版权审查等|
|511|网络接入认证|WiFi Portal 等，非源站业务码|
|422|名称更新|从 "Unprocessable Entity"→"Unprocessable Content"（RFC 9110）|

## 2.5 少用 / 特定协议扩展

|码|协议|说明|
|---|---|---|
|423 / 424|WebDAV|资源锁定 / 依赖操作失败|
|507 / 508|WebDAV|存储不足 / 循环检测|
|501|HTTP|服务端不支持该功能/方法（≠业务未开发）|
|505|HTTP|协议版本不支持|
|418|—|I'm a teapot，**不要用于正式 API**|

> [!tip] 思辨：418 为什么不能用？

> 418 源自 RFC 2324（Hyper Text Coffee Pot Control Protocol），是 1998 年愚人节的玩笑。IANA 注册表中它被标记为保留但未正式定义语义。在生产 API 中使用它会导致监控系统无法正确分类错误类型，客户端 SDK 也可能不识别。
