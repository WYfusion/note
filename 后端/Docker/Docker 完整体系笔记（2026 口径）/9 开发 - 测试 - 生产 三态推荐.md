## 0. 定位

> 开发态、测试态/CI、生产态三种环境的具体配置差异与推荐实践。本页覆盖 **Docker 三态工作流的完整方案**。

---

## 1. 三态对比总览

|维度|开发态|测试态 / CI|生产态|
|---|---|---|---|
|**源码挂载**|bind mount|镜像内包含|镜像内包含|
|**热重载**|`docker compose watch`|不需要|不需要|
|**镜像标签**|`latest` 可接受|CI 构建的确定 tag|固定 tag / digest|
|**端口暴露**|`127.0.0.1` • debug 端口|最小化|仅入口层|
|**安全配置**|宽松（方便调试）|中等|最严格（非 root / 只读 / 最小权限）|
|**依赖服务**|Compose 本地拉起|Compose 拉起 + 一次性容器|独立管理或托管|
|**healthcheck**|可选|推荐|必须|
|**restart 策略**|no / unless-stopped|no|unless-stopped / always|
|**资源限制**|无|可选|明确设置|

## 2. 开发态

> [!important]
> 
> 开发态的核心目标是：**快速迭代**。代码修改后应立即反映到容器中，调试工具随时可用，依赖服务一键拉起。

```YAML
# compose.dev.yaml
services:
  api:
    build:
      context: ./services/api
    volumes:
      - ./services/api/src:/app/src       # bind mount 源码
    ports:
      - "127.0.0.1:8000:8000"             # 仅本机
      - "127.0.0.1:5678:5678"             # debug 端口
    environment:
      - LOG_LEVEL=debug
    develop:
      watch:
        - action: sync
          path: ./services/api/src
          target: /app/src
        - action: rebuild
          path: ./services/api/requirements.txt

  db:
    image: postgres:16-alpine
    ports:
      - "127.0.0.1:5432:5432"             # 本地调试用
    volumes:
      - db-data:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine
```

**开发态核心实践**：

- ✅ bind mount 源码——编辑即时生效

- ✅ `docker compose watch`——自动同步/重建

- ✅ debug 端口仅绑定 `127.0.0.1`

- ✅ DB/缓存用 Compose 本地拉起

- ✅ 可加开发工具容器（lint/test/shell）

## 3. 测试态 / CI

> [!important]
> 
> 测试态的核心目标是：**可复现、可靠、快速**。用 Compose 拉起依赖服务，以一次性容器跑测试，尽量复用生产镜像构建链。

```Bash
# CI 中的典型流程

# 1. 构建镜像（复用生产 Dockerfile）
docker buildx bake --set "*.cache-from=type=registry,ref=myregistry.com/cache"

# 2. 拉起依赖服务
docker compose -f compose.yaml up -d db redis

# 3. 等待依赖就绪
docker compose -f compose.yaml exec db pg_isready -U postgres

# 4. 跑测试（一次性容器）
docker compose -f compose.yaml run --rm api pytest

# 5. 清理
docker compose -f compose.yaml down -v
```

**测试态核心实践**：

- ✅ 用 Compose 拉起依赖服务（DB、Redis、消息队列）

- ✅ 以一次性容器（`run --rm`）跑测试

- ✅ 尽量复用生产镜像构建链——确保测的和部的一致

- ✅ Build cache 与 registry cache 结合——加速 CI 构建

- ✅ 测试完成后清理环境（`down -v`）

## 4. 生产态

> [!important]
> 
> 生产态的核心目标是：**稳定、安全、可观测**。所有配置应明确、可复现，安全基线必须到位。

```YAML
# compose.prod.yaml
x-security: &security
  read_only: true
  user: "1000:1000"
  cap_drop: [ALL]
  security_opt: ["no-new-privileges:true"]
  tmpfs: [/tmp, /run]

services:
  api:
    <<: *security
    image: myregistry.com/myapp-api:v1.2.3    # 固定 tag
    ports:
      - "127.0.0.1:8000:8000"                  # 仅入口层
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 10s
      timeout: 5s
      retries: 3
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: "2.0"
          memory: 2G
    secrets:
      - db-password
      - api-key

  db:
    <<: *security
    image: postgres:16-alpine
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      retries: 5
    restart: unless-stopped
    volumes:
      - db-data:/var/lib/postgresql/data
    # ❌ 生产不暴露端口
```

**生产态核心实践**：

- ✅ 不 bind mount 源码——镜像包含所有代码

- ✅ 使用固定镜像 tag / digest——可复现部署

- ✅ 只暴露入口层端口（gateway / API）

- ✅ 明确 healthcheck / restart / resource limits / secrets

- ✅ 非 root / 只读根文件系统 / 最小权限

- ✅ 监控、日志、备份单独设计
