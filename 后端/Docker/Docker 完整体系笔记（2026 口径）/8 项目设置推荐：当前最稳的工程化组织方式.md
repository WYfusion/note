## 1. 推荐目录结构

> [!important]
> 
> **工程判断**：以下是经过实践验证的推荐目录结构。核心原则：**关注点分离**——服务代码、基础设施配置、运维脚本、机密管理各自独立。

```Plain
project/
├── compose.yaml            # 基础服务拓扑（单一真相源）
├── compose.dev.yaml        # 开发态覆盖（bind mount / watch / debug）
├── compose.prod.yaml       # 生产态覆盖（安全加固 / 资源限制）
├── compose.gpu.yaml        # GPU 服务覆盖
├── .env                    # 当前环境变量（不提交 Git）
├── .env.example            # 环境变量模板（只放键名与示例）
├── .dockerignore           # 构建上下文排除
├── docker-bake.hcl         # 多镜像构建编排（可选）
├── services/               # 服务代码 + Dockerfile
│   ├── api/
│   │   ├── Dockerfile
│   │   └── src/
│   ├── worker/
│   │   ├── Dockerfile
│   │   └── src/
│   └── web/
│       ├── Dockerfile
│       └── src/
├── infra/                  # 基础设施配置
│   ├── nginx/ or caddy/
│   ├── db/
│   └── observability/
├── secrets/                # 机密文件（仅本地开发）
├── scripts/                # 辅助脚本
└── Makefile                # 任务入口
```

## 2. Compose 文件组织推荐

### 2.1 文件职责分工

|文件|职责|关键内容|
|---|---|---|
|`compose.yaml`|**基础服务拓扑**（单一真相源）|所有服务定义、网络、卷、基本配置|
|`compose.dev.yaml`|**开发态覆盖**|bind mount、watch、debug 端口、mock 服务|
|`compose.prod.yaml`|**生产态覆盖**|安全加固、固定 tag、只保留必要 ports、resource limits|
|`compose.gpu.yaml`|**GPU 服务覆盖**|GPU 设备声明、模型缓存、推理服务|

### 2.2 合并命令示例

```Bash
# 开发环境
docker compose -f compose.yaml -f compose.dev.yaml up

# 生产环境
docker compose -f compose.yaml -f compose.prod.yaml up -d

# GPU 开发
docker compose -f compose.yaml -f compose.dev.yaml -f compose.gpu.yaml up
```

### 2.3 Makefile 封装

```Makefile
# Makefile
.PHONY: dev prod gpu test

dev:
	docker compose -f compose.yaml -f compose.dev.yaml up

prod:
	docker compose -f compose.yaml -f compose.prod.yaml up -d

gpu:
	docker compose -f compose.yaml -f compose.dev.yaml -f compose.gpu.yaml up

test:
	docker compose -f compose.yaml -f compose.dev.yaml run --rm api pytest

build:
	docker buildx bake
```

## 3. 环境变量组织推荐

### 3.1 文件分工

|文件|作用|是否提交 Git|
|---|---|---|
|`.env.example`|只放键名与示例值，作为模板|✅ 提交|
|`.env`|当前环境的实际值|❌ 不提交|
|`.env.local`|本地开发特有配置|❌ 不提交|
|`.env.prod`|生产环境配置|❌ 不提交（或加密存储）|

### 3.2 原则

- ✅ **运行期配置与构建期参数分离**——构建期用 `ARG`，运行期用 `environment` / `env_file`

- ✅ 高敏信息不要直接提交 Git——用 Compose `secrets` 或外部 secret 管理

- ✅ `.env.example` 始终保持更新——新成员 clone 项目后可立即知道需要哪些变量

```Plain
# .env.example
POSTGRES_USER=postgres
POSTGRES_PASSWORD=      # 填写你的密码
POSTGRES_DB=app
API_PORT=8000
LOG_LEVEL=info
MODEL_CACHE_DIR=/models
```

## 4. 数据与状态目录推荐

|数据类型|推荐挂载方式|说明|
|---|---|---|
|**DB 数据**|volume|PostgreSQL / MySQL / Redis 数据目录|
|**缓存**|volume 或 tmpfs|视是否需要持久化|
|**上传文件**|volume|用户生成内容|
|**源代码**|bind mount（仅开发）|生产不挂载源码|
|**模型缓存**|volume|避免每次重新下载|
|**日志**|stdout/stderr + 外部日志系统|避免容器内乱写文件|
|**机密**|Compose secrets|不通过环境变量传递|

> [!important]
> 
> **常见误区：日志写入容器内文件**
> 
> Docker 的推荐实践是应用日志输出到 **stdout/stderr**，由 Docker 日志驱动（Log Driver）收集并转发到外部日志系统（如 Loki、ELK、CloudWatch）。不要在容器内写日志文件，除非有明确的历史原因。
