## 1. 为什么安全是最容易被忽视的一层

> [!important]
> 
> Docker 提供的是**进程级隔离**，不是虚拟机级隔离。容器内的 root 用户默认与宿主机的 root 是**同一个 UID（0）**。如果容器逃逸（Container Escape）成功，攻击者就是宿主机 root。因此，Docker 安全的核心思路是：**缩小攻击面、降低权限、减少泄露**。

## 2. 运行时安全基线

### 2.1 非 root 运行

> [!important]
> 
> **非 root 运行**是指容器内的主进程以**非 root 用户（UID ≠ 0）**执行。即使容器被突破，攻击者获得的也仅是低权限用户的能力。

```Docker
# Dockerfile 中设置非 root 用户
RUN groupadd -r appuser && useradd -r -g appuser appuser
USER appuser
```

```YAML
# Compose 中指定用户
services:
  api:
    user: "1000:1000"
```

### 2.2 只读根文件系统

> [!important]
> 
> **只读根文件系统（Read-only Root Filesystem）** 将容器的根文件系统设为只读。攻击者无法在容器内写入恶意文件、篡改二进制或植入后门。需要写入的路径通过 volume 或 tmpfs 单独提供。

```YAML
services:
  api:
    read_only: true
    tmpfs:
      - /tmp
      - /run
    volumes:
      - app-data:/app/data    # 只有明确需要写入的路径才挂载
```

### 2.3 Linux Capabilities 控制

> [!important]
> 
> **Linux Capabilities（能力）** 是将传统 root 的「超级权限」拆分成多个独立的细粒度权限单元。例如 `NET_RAW`（原始套接字）、`SYS_ADMIN`（系统管理）、`NET_BIND_SERVICE`（绑定 1024 以下端口）等。Docker 默认保留了部分 capabilities，但安全最佳实践是：**先移除全部，再按需加回**。

```YAML
services:
  api:
    cap_drop:
      - ALL               # 移除所有 capabilities
    cap_add:
      - NET_BIND_SERVICE  # 仅按需加回
```

### 2.4 禁止提权

```YAML
services:
  api:
    security_opt:
      - no-new-privileges:true   # 禁止容器内进程通过 setuid/setgid 等机制获取新权限
```

### 2.5 完整安全基线模板

> [!important]
> 
> **工程判断**：以下是每个生产服务应当采用的最小安全基线。可通过 Compose 的 YAML 锚点统一应用到所有服务。

```YAML
# 安全基线可复用片段
x-security-baseline: &security-baseline
  read_only: true
  user: "1000:1000"
  cap_drop: [ALL]
  security_opt: ["no-new-privileges:true"]
  tmpfs:
    - /tmp
    - /run

services:
  api:
    <<: *security-baseline
    image: myapp-api:latest
    # 如果需要额外可写路径，单独加 volume
    volumes:
      - app-uploads:/app/uploads
```

### 2.6 应避免的危险操作

|危险操作|风险|替代方案|
|---|---|---|
|`privileged: true`|容器获得几乎等同宿主机 root 的权限|按需 `cap_add` 特定 capabilities|
|挂载 Docker socket|容器可控制宿主机上所有容器|使用受控代理或专用 API|
|大面积 bind mount 宿主机目录|容器可读写宿主机关键文件|最小化挂载范围 + `:ro`|
|root 用户运行|逃逸后直接是宿主机 root|`USER non-root` • `user: "1000:1000"`|

## 3. Rootless 与 userns-remap

### 3.1 Rootless Docker

> [!important]
> 
> **Rootless Docker（无 root Docker）** 是指 Docker daemon 本身以**非 root 用户**运行。传统模式下 `dockerd` 以 root 运行，意味着 Docker 本身就是一个高权限服务。Rootless 模式将整个 Docker 运行在用户命名空间中，即使 Docker 本身被突破，攻击者也仅获得普通用户权限。

**优势**：

- 安全收益最高——从根源消除 root daemon 风险

- 适合多用户共享服务器

**限制**：

- 部分功能受限（如特定网络模式、某些存储驱动）

- 性能可能略有影响

### 3.2 userns-remap（用户命名空间重映射）

> [!important]
> 
> **userns-remap（用户命名空间重映射）** 是在保持传统 root-daemon 架构的同时，将容器内的 root（UID 0）映射为宿主机上的一个**非特权 UID**。容器内看起来是 root，但实际上在宿主机上没有 root 权限。

### 3.3 对比

|维度|Rootless Docker|userns-remap|
|---|---|---|
|**daemon 权限**|非 root|root（传统模式）|
|**容器内 root 映射**|映射为宿主机非特权用户|映射为宿主机非特权用户|
|**安全收益**|⭐⭐⭐⭐⭐ 最高|⭐⭐⭐ 中等|
|**兼容性**|部分功能受限|兼容性更好|
|**适用场景**|高安全诉求环境|需要平滑过渡的现有环境|

> [!important]
> 
> **工程判断**：高安全诉求优先评估 Rootless。如果环境约束不允许，userns-remap 是次优选择。两者都不影响容器内的非 root 用户设置——它们是不同层次的防御。

## 4. 机密管理全链路

### 4.1 构建期机密

> [!important]
> 
> 构建期机密是指在 `docker build` 过程中临时需要的敏感信息（私有包仓库 token、云凭证、SSH 密钥等）。它们必须**不出现在任何镜像层中**。

```Docker
# ✅ 正确：使用 BuildKit secret mount
RUN --mount=type=secret,id=pypi_token \
    pip install --index-url https://$(cat /run/secrets/pypi_token)@private.pypi.com/simple/ -r requirements.txt

# ❌ 错误：绝对不要这样做
ARG PYPI_TOKEN=xxx
RUN pip install --index-url https://${PYPI_TOKEN}@private.pypi.com/simple/ -r requirements.txt
# ↑ PYPI_TOKEN 会出现在 docker history 中
```

### 4.2 运行期机密

```YAML
# Compose secrets
secrets:
  db-password:
    file: ./secrets/db-password.txt   # 本地开发用文件
  # 生产环境可用外部 secret 管理系统

services:
  db:
    image: postgres:16-alpine
    secrets:
      - db-password
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db-password
```

### 4.3 机密管理决策树

![[6 安全：当前项目设置中最关键的一层 - 4.3 机密管理决策树 - 图 02 .excalidraw|800]]

|方式|安全性|适用场景|风险|
|---|---|---|---|
|`ARG` / `ENV` 写入 Dockerfile|❌ 极低|**绝不用于机密**|值存储在镜像层/元数据中|
|`--mount=type=secret`|✅ 高|构建期机密|仅构建期可见，不进入镜像|
|`.env` • `environment`|⚠️ 中等|低敏运行时配置|不要提交 Git；容器内可见|
|Compose `secrets`|✅ 高|运行期高敏机密|以文件形式挂载，不进环境变量|

## 5. 远程 daemon 安全

> [!important]
> 
> **裸 TCP 暴露 Docker daemon 极其危险**
> 
> Docker daemon API 拥有宿主机的完整控制权。裸暴露 TCP 2375/2376 端口等同于把 root 权限开放给网络上的任何人。

**必须遵守的原则**：

- ✅ 通过 **SSH 隧道**访问远程 daemon（推荐）

- ✅ 通过 **TLS 双向认证**访问

- ✅ 通过受控代理（如专用管理工具）访问

- ❌ **绝不**把 daemon 端口直接暴露到内网/公网

- ❌ **绝不**把这当作「方便开发」的手段

```Bash
# ✅ 推荐：通过 SSH 访问远程 daemon
docker context create remote --docker "host=ssh://user@remote-host"
docker context use remote
```

## 6. 防火墙与网络规则

> [!important]
> 
> **「换更底层网络模式 = 自己承担更多安全责任」**
> 
> - **bridge 网络**：Docker 自动创建 iptables 规则，提供基本的网络隔离和端口转发
> 
> - **host 网络**：Docker 不创建任何规则，容器直接使用宿主机网络栈
> 
> - **macvlan / ipvlan**：Docker 不提供防火墙规则兜底
> 
> 因此，从 bridge 切换到 host、macvlan、ipvlan 意味着你需要**自己配置防火墙规则和网络控制**。

## 安全检查清单

- [ ] 容器以非 root 用户运行

- [ ] `read_only: true` + 必要的 tmpfs/volume

- [ ] `cap_drop: [ALL]` + 按需 `cap_add`

- [ ] `security_opt: ["no-new-privileges:true"]`

- [ ] 无 `privileged: true`

- [ ] 未挂载 Docker socket

- [ ] 未大面积 bind mount 宿主机关键目录

- [ ] 机密未通过 ARG/ENV 写入 Dockerfile

- [ ] 构建期机密使用 `--mount=type=secret`

- [ ] 运行期机密使用 Compose `secrets`

- [ ] 只暴露必要端口，本机绑定 `127.0.0.1`

- [ ] 远程 daemon 通过 SSH/TLS 访问

- [ ] 基础镜像使用固定版本标签
