## 1. Docker 解决的核心问题

> [!important]
> 
> **直觉类比**：Docker 像一个「标准集装箱」——不管里面装的是什么货物（代码 + 依赖），只要符合集装箱规格，就能在任何码头（服务器）上装卸运行，无需关心码头的具体设施差异。

Docker 的价值不在于「虚拟化」，而在于解决四个工程痛点：

### 1.1 构建一致性（Build Consistency）

**是什么**：将「代码 + 运行时（Runtime） + 系统依赖（System Dependencies） + 启动命令」打包为一个不可变的**镜像（Image）**。

**为什么重要**：消灭「我本地能跑，服务器上跑不了」的经典问题。根本原因在于：本地与服务器的操作系统版本、库版本、环境变量、文件路径往往不同。Docker 把这些差异全部封装进镜像，使构建产物与宿主机解耦。

### 1.2 运行隔离（Runtime Isolation）

**是什么**：每个容器拥有独立的文件系统视图、网络栈（Network Stack）、进程树（Process Tree）和权限边界。

**为什么重要**：多个服务可以在同一台物理机上互不干扰地运行——不同版本的 Python、不同端口需求、不同系统库互不冲突。

**怎么工作**：Docker 利用 Linux 内核的三个核心机制实现隔离：

|内核机制|隔离维度|效果|
|---|---|---|
|**命名空间（Namespace）**|PID / NET / MNT / UTS / IPC / USER|每个容器看到独立的进程树、网络、文件系统、主机名|
|**控制组（Cgroup）**|CPU / 内存 / IO / 网络带宽|限制和隔离资源使用量，防止单容器吃光资源|
|**联合文件系统（UnionFS）**|文件系统层|镜像分层只读 + 容器可写层，高效共享与隔离|

> [!important]
> 
> **常见误区：「Docker 容器 = 轻量虚拟机」**
> 
> 容器**不是**虚拟机。虚拟机（VM）通过 Hypervisor 模拟完整硬件，运行独立内核；容器**共享宿主机内核**，仅通过命名空间和 Cgroup 实现进程级隔离。因此容器启动快、开销小，但隔离强度低于 VM。

### 1.3 交付一致性（Delivery Consistency）

**是什么**：本地开发、CI 构建、测试环境、生产服务器运行**同一个镜像**（或同一构建链产出的镜像）。

**为什么重要**：传统部署依赖手工操作或脚本，环境漂移（Environment Drift）是线上故障的重大来源。镜像一旦构建完成，其内容就是确定的，不会因为部署目标不同而变化。

### 1.4 多服务编排（Multi-service Orchestration）

**是什么**：真实项目不是单个容器能解决的——Web 前端、API 后端、数据库、缓存、消息队列、模型服务……需要一套声明式的方式定义它们的拓扑、依赖、网络和存储。

**为什么重要**：Docker Compose 正是为此而生——用一个 `compose.yaml` 文件声明所有服务及其关系，一条命令启动整个项目。

> [!important]
> 
> **工程判断**：单容器是 Docker 的入门点，但 Compose 才是现代本地开发和中小部署的真正主力。不要停留在单容器阶段。

## 2. 四个最核心对象

> [!important]
> 
> 记住这个公式：**Dockerfile → Image → Container + Volume + Network**。Dockerfile 是输入，Image 是产物，Container 是运行态，Volume 和 Network 是运行态的两个核心扩展维度。

### 2.1 镜像（Image）

**定义**：镜像是一个**只读的、分层的文件系统快照**，附带元数据（环境变量、启动命令、暴露端口等）。它是构建（`docker build`）的产物，也是运行容器的模板。

**核心机制**——分层存储（Layered Storage）：

![[2026-04-18 20.36.05docker分层存储.excalidraw|200]]

每条 Dockerfile 指令生成一层；层与层之间通过**内容寻址（Content Addressable Storage）**去重——如果两个镜像共享前三层，磁盘上只存一份。

**关键认知三条**：

1. 镜像应尽量**小、确定、可复现**——使用固定 tag 或 digest，避免 `latest`

1. **不应把运行态数据写进镜像**——数据库文件、用户上传、日志不进镜像

1. **不应把机密写进镜像层**——密钥、token、证书不能出现在任何 `COPY`、`ENV`、`ARG` 中

### 2.2 容器（Container）

**定义**：容器是镜像的**运行实例**——在镜像的只读层之上叠加一个**可写层（Writable Layer）**，加上命名空间和 Cgroup 隔离，形成一个独立的运行环境。

**生命周期**：

![[2026-04-18 20.36.46docker容器.excalidraw|600]]

**关键认知三条**：

1. **容器不是虚拟机**——不要期待「长期 SSH 进去手工维护」

1. **容器应可删除、可重建**——任何手工修改都会在 `docker rm` 后丢失

1. **配置和数据必须外置**——通过 Volume、Bind Mount、环境变量、Secret 注入

> [!important]
> 
> **常见误区：「容器里装了那么多东西，删了好可惜」**
> 
> 如果删除容器让你感到不安，说明你把不该放在容器里的东西放进去了。正确做法：代码在镜像里、数据在 Volume 里、配置在环境变量/Secret 里。容器本身是**一次性计算单元**。

### 2.3 卷与挂载（Volume / Mount）

**定义**：Volume 和 Mount 是 Docker 提供的**数据持久化与共享机制**——让数据的生命周期独立于容器。

**三种类型速览**（详见 §4 专题页面）：

|类型|管理者|典型场景|与宿主机耦合度|
|---|---|---|---|
|**Volume**|Docker Engine|数据库文件、模型缓存、索引|低|
|**Bind Mount**|用户手动指定宿主机路径|开发态源码热更新、配置文件|高|
|**tmpfs**|内存|临时敏感文件、高速临时缓存|无（纯内存）|

### 2.4 网络（Network）

**定义**：Docker 网络定义了容器之间、容器与宿主机、容器与外部世界的**通信边界与规则**。

**核心判断**（详见 §5 专题页面）：

|场景|推荐网络模式|一句话理由|
|---|---|---|
|单机多容器（99% 场景）|**用户自定义 bridge**|隔离清晰、服务发现自然、端口控制灵活|
|多机容器互联|**overlay**|跨主机分布式网络|
|需要宿主机网络栈|**host**|少一层 NAT，仅限特殊需求|
|完全断网|**none**|离线计算、高隔离沙箱|

## 3. 当前推荐总原则（2026 口径）

以下是当前 Docker 生态的**正确默认值**——除非有明确理由，否则按此执行：

|维度|正确默认值|常见错误做法|
|---|---|---|
|**单服务构建**|Dockerfile + 多阶段构建|单阶段巨型镜像|
|**多服务编排**|`compose.yaml`（Compose Specification）|shell 脚本串起 `docker run`|
|**数据持久化**|Volume 优先|数据写在容器可写层|
|**开发源码**|Bind Mount|每次改代码重新 build|
|**网络**|用户自定义 bridge|默认 bridge 或随意 host|
|**端口暴露**|最少暴露，入口层才 publish|所有服务 `0.0.0.0:xxx`|
|**安全**|非 root / read_only / cap_drop: ALL|`privileged: true` 解决一切|
|**构建工具**|BuildKit（多阶段 + cache mount + secret mount）|传统 builder、不用缓存|
|**机密管理**|`RUN --mount=type=secret` • Compose secrets|`ARG` / `ENV` 传密钥进镜像|
|**配置文件名**|`compose.yaml`|`docker-compose.yml`（旧习惯）|
