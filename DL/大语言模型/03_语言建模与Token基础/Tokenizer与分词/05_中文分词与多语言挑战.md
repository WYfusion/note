# 中文分词与多语言挑战

不同语言的特性差异巨大，给 Tokenizer 的设计带来了独特的挑战。中文作为一种非拉丁语系的代表，具有显著的特点。

## 1. 中文分词的特殊性

### 1.1 字 vs 词
*   **英文**: 单词之间有天然的空格分隔，Token 通常是单词或词根。
*   **中文**: 汉字之间没有空格，且“词”的定义模糊（如“大语言模型”是 1 个词还是 3 个词？）。

### 1.2 两种流派
1.  **基于词 (Word-based)**: 先用 Jieba 等工具分词，再做 BPE。
    *   *优点*: 语义完整。
    *   *缺点*: 词表极大，OOV 问题严重。
2.  **基于字 (Character-based)**: 每个汉字作为一个 Token。
    *   *优点*: 词表小，无 OOV。
    *   *缺点*: 序列太长，语义被割裂（“苹果”分开看只是两个字）。
3.  **基于子词 (Subword-based, BPE/SentencePiece)**: **主流方案**。
    *   不依赖分词工具，直接在生语料上跑 BPE。
    *   高频词（“的”、“是”、“中国”）会成为独立 Token。
    *   低频词会被拆分为字。

## 2. 多语言模型的挑战 (Multilingual)

当一个模型需要同时支持英文、中文、代码等多种语言时，Tokenizer 需要平衡各方利益。

### 2.1 压缩率不平衡
*   如果训练语料中英文占 90%，中文占 10%，BPE 会倾向于合并英文子词。
*   **结果**: 一个英文单词往往是 1 个 Token，而一个中文词可能被拆成 3 个字节 Token（UTF-8 编码下，一个汉字占 3 字节）。
*   **影响**: 中文推理成本变高（序列变长），且效果变差。

### 2.2 解决方案
1.  **扩大词表**: 如 Qwen 使用了 150k 词表，专门加入了大量中文字词，使得中文压缩率大幅提升（平均 1 个汉字 < 1 Token）。
2.  **重采样 (Up-sampling)**: 在训练 Tokenizer 时，人为增加低资源语言的语料权重。

## 3. 数字与代码的处理

### 3.1 数字分割
*   **问题**: 早期 Tokenizer 将 "12345" 作为一个 Token。模型无法理解 "12345" 和 "12346" 的数值关系，也无法进行加减运算。
*   **现代做法**: 将数字按位切分（Digit-by-digit）。
    *   Llama 2: "123" -> "1", "2", "3"。
    *   这大大增强了模型的数学算术能力。

### 3.2 代码中的空格
*   Python 依赖缩进。Tokenizer 必须精确保留空格和 Tab。
*   SentencePiece 将空格视为特殊字符 `_` (U+2581)，确保可逆还原。

## 4. 常见模型对比

| 模型 | 词表大小 | 中文处理 | 特点 |
| :--- | :--- | :--- | :--- |
| **GPT-4 (cl100k)** | ~100k | 一般 | 字节级 BPE，中文通常 1-3 Token |
| **Llama 3** | 128k | 较好 | 相比 Llama 2 大幅扩充词表，多语言能力提升 |
| **Qwen (千问)** | ~152k | **极好** | 针对中文深度优化，压缩率极高 |
| **Baichuan** | ~64k | 好 | 针对中文优化 |

## 5. 总结
评价一个 Tokenizer 对某种语言好坏的核心指标是 **压缩率 (Compression Ratio)**：
$$ \text{Ratio} = \frac{\text{Bytes of Text}}{\text{Number of Tokens}} $$
比值越大，说明每个 Token 承载的信息越多，推理效率越高。
