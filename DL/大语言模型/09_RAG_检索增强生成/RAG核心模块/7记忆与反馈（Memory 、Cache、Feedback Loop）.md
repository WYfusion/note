## 是什么
利用历史交互记录、缓存机制和用户反馈来提升 RAG 系统的效率、连贯性和持续改进能力。

## 三大子模块
### Memory（对话记忆）
在多轮对话中维护上下文连贯性。
- **Short-term Memory**：当前对话的历史消息
- **Long-term Memory**：跨会话的用户偏好、历史交互
- **作用**：让 query rewrite 能利用历史上下文（例如指代消解：「它」指的是什么）

### Cache / Semantic Cache（语义缓存）
缓存已处理过的查询和结果，避免重复计算。
- **Exact Cache**：完全相同的查询直接返回缓存结果
- **Semantic Cache**：语义相似的查询复用结果（用向量相似度判断）
- **收益**：降低延迟、减少 API 成本、提升吞吐量
- **挑战**：缓存失效策略（知识库更新后旧缓存可能不准）

### Feedback Loop（反馈闭环）
通过用户反馈和系统日志持续优化 RAG 系统。
- **显式反馈**：用户点赞/点踩、纠正错误
- **隐式反馈**：点击率、停留时间、二次检索率
- **优化方向**：
    - 调整检索排序权重
    - 优化切块策略
    - 更新嵌入模型
    - 重建索引

## 反馈驱动的优化流程
`日志收集 → 评测指标计算 → 问题诊断 → 参数调优 → 重建索引 → 验证效果`

## 与其他模块的关系
- **Memory** → 为 Query Reformulation 提供历史上下文
- **Cache** → 在 Retrieval 前拦截，直接返回结果
- **Feedback** → 影响所有模块的持续优化
