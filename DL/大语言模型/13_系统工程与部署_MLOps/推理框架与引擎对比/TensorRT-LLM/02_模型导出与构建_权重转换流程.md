# 模型导出与构建：权重转换流程

TensorRT-LLM 的核心工作流是将 PyTorch 模型转换为高度优化的 TensorRT Engine。这个过程比 vLLM 复杂，需要显式的“编译”步骤。

## 1. 标准构建流程

流程通常分为三步：
1.  **下载模型**: 获取 HuggingFace 格式的权重。
2.  **权重转换 (Convert)**: 将 HF 权重转换为 TensorRT-LLM 的中间格式（通常是 safetensors + config）。
3.  **构建引擎 (Build)**: 针对特定 GPU 架构编译 Engine 文件。

### 1.1 权重转换 (Checkpoint Conversion)
使用 `trtllm-build` 之前的准备步骤。

```bash
# 示例：转换 Qwen 模型
python3 convert_checkpoint.py \
    --model_dir ./Qwen-Audio-Chat \
    --output_dir ./tllm_checkpoint \
    --dtype float16 \
    --tp_size 1  # 单卡推理
```

### 1.2 构建引擎 (Build Engine)
这一步会进行算子融合和内核选择。

```bash
trtllm-build \
    --checkpoint_dir ./tllm_checkpoint \
    --output_dir ./tllm_engines \
    --gemm_plugin float16 \
    --max_batch_size 8 \
    --max_input_len 2048 \
    --max_output_len 1024
```

*   **注意**: 构建出的 Engine 是**硬件绑定**的。在 A100 上构建的 Engine 无法在 4090 上运行。

## 2. 语音大模型 (Whisper) 特例

Whisper 的结构包含 Encoder 和 Decoder，处理流程略有不同。

### 2.1 构建 Encoder
Encoder 通常是标准的 Transformer Encoder，可以直接转为 TensorRT Engine。

```bash
# 1. 导出 ONNX
python3 export_onnx.py --model large-v3 --output_dir ./onnx_models

# 2. 构建 TRT Engine
trtexec --onnx=./onnx_models/encoder.onnx \
        --saveEngine=./engines/encoder.plan \
        --fp16
```

### 2.2 构建 Decoder
Decoder 是自回归模型，使用 TensorRT-LLM 构建。

```bash
# 使用 TensorRT-LLM 提供的构建脚本
python3 build.py \
    --model_name large-v3 \
    --output_dir ./engines \
    --use_gpt_attention_plugin float16 \
    --enable_context_fmha
```

---

## 3. 关键参数详解

在构建 Audio LLM 时，以下参数至关重要：

*   **`--max_input_len`**:
    *   对于文本模型，这是 Token 数。
    *   对于 Audio 模型，这通常指 Audio Feature 的长度（例如 Whisper 的 3000 帧）。
    *   **坑**: 如果音频过长，超过此限制会直接报错，需预先切分。

*   **`--max_batch_size`**:
    *   决定了显存预分配的大小。设置过大会浪费显存，设置过小会限制吞吐。

*   **`--paged_kv_cache`**:
    *   必须开启，以支持类似 vLLM 的显存管理，避免长音频生成的碎片化。

## 4. 验证构建结果

构建完成后，目录中通常包含：
*   `rank0.engine`: 主引擎文件。
*   `config.json`: 包含模型结构和元数据。

可以使用 `run.py` 脚本进行简单测试：

```bash
python3 ../run.py --engine_dir ./engines --input_text "测试音频转录"
```
