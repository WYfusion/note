# 9. 训练崩溃救援

训练崩溃 = loss 爆炸 / NaN / 进程挂起。这是最紧急的情况，需要标准化的应急流程。

---

## 立即动作（前 5 分钟）

1. **停在最近安全 checkpoint**（不要继续跑）
2. **保存异常 batch**（元信息 + 原始数据）
3. **保留 stdout/stderr + 每个 rank 的日志**

---

## 修复优先级

按成本从低到高排序：

| 优先级 | 动作 | 目标 |
| --- | --- | --- |
| P0 | 查 nonfinite / overflow | 确认是否数值问题 |
| P1 | 降 LR | 减小 update 幅度 |
| P2 | 加长 warmup | 稳定 optimizer 状态 |
| P3 | 启用 / 收紧 grad clipping | 限制梯度爆炸 |
| P4 | 缩短 max seq len | 降低单 batch 风险 |
| P5 | 排查坏样本 | 消除数据源异常 |
| P6 | 关闭可疑 fused kernel / compile | 排除实现 bug |
| P7 | 核对 resume 状态 | 确认 checkpoint 完整 |

---

## 混合精度专项

<aside>
⚙️

- **fp16**：第一步看 `loss scale`——如果 scale 持续下降到极小值，说明频繁 overflow
- **bf16**：动态范围更大，能用 bf16 **尽量用 bf16**，可避免大部分 overflow 问题
- **fp8**：需额外监控 `amax`（激活最大值）和 `scaling factor`，确保量化不失真
</aside>
