## 一句话概括

RRF（倒数排名融合）是最常用的多路召回融合算法，只看排名不看分数，天然解决不同检索方式分数尺度不一致的问题。简单、鲁棒、无需调参，开箱即用。

## 核心原理

### 问题背景

Hybrid Retrieval 同时用 BM25 和 Dense 检索，但两路的分数尺度完全不同：

- BM25 分数可能是 15.3
- Dense cosine 相似度可能是 0.87

没法直接比较或相加，怎么办？

### RRF 的解决思路

**不看分数，只看排名**。不管分数是多少，只关心文档在每路检索结果中排第几。

### 数学公式

$RRF\_score(d) = \sum_{r \in R} \frac{1}{k + rank_r(d)}$

- $d$：文档
- $R$：所有检索路（如 BM25、Dense）
- $rank_r(d)$：文档 $d$ 在第 $r$ 路检索中的排名（1-based）
- $k$：平滑参数（原始论文推荐 60）

### 逐步计算举例

假设 BM25 和 Dense 两路召回，$k = 60$：

|文档|BM25 排名|Dense 排名|RRF 分数|
|---|---|---|---|
|Doc B|5|1|1/65 + 1/61 = 0.0154 + 0.0164 = 0.0318|
|Doc D|-|2|0 + 1/62 = 0.0161|

结果：Doc A 最终排第 1，因为它在两路中都排名靠前。

### 参数 k 的作用

- $k$ 越大 → 不同排名之间的分数差距越小 → 更“平均”
- $k$ 越小 → 排名靠前的文档优势越明显
- $k = 60$ 是原始论文推荐值，实践中几乎不需要调

### 处理缺失排名

如果一篇文档只在一路中被召回，另一路没有它：

- 只计算有排名的那路的分数
- 缺失的路贡献 0 分
- 这意味着在多路中都被召回的文档天然得分更高

## 为什么 RRF 如此流行？

1. **无需归一化**：不用处理分数尺度不一致的问题
2. **无需调参**：k=60 几乎在所有场景都工作得很好
3. **无需训练**：纯算法，不依赖任何训练数据
4. **鲁棒性好**：对各路检索质量的波动不敏感
5. **可扩展**：轻松支持 2 路、3 路、N 路融合

## 与 Weighted Sum 的对比

|特性|RRF|Weighted Sum|
|---|---|---|
|归一化|不需要|必须归一化分数|
|各路权重|相同（无法偏重）|可调整（灵活）|
|复杂度|极低|低|

## 局限

- **无法利用分数信息**：排名第 1 和第 2 的分数差可能很大，但 RRF 看不到
- **各路权重相同**：无法指定某路更重要（比如希望 Dense 权重更高）
- **对召回数量敏感**：各路召回数量差异大时可能有偏差

## 工程实现

- **Elasticsearch 8.x**：原生支持 RRF 查询
- **Weaviate**：内置 Hybrid Search（BM25 + Dense + RRF）
- **LangChain**：`EnsembleRetriever` 支持 RRF
- **LlamaIndex**：`QueryFusionRetriever` 支持
- **自实现**：代码极简，10 行即可
