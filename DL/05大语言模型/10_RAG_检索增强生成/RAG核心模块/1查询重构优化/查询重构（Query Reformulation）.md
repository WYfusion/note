## 是什么
将用户的原始问题转化为更适合检索的查询形式。用户的提问往往模糊、口语化或包含歧义，直接检索效果差，需要先「翻译」成检索引擎能理解的语言。

## 为什么重要
- 用户问题和文档表述之间存在**语义鸿沟**
- 好的查询重构能显著提升召回率和准确率
- 是整个 RAG 流水线的**第一步**，直接影响后续所有环节

## 常用技术
### Query Rewrite（查询改写）
用 LLM 将口语化问题改写为更精确的检索查询。

`"这个月的销售怎么样" → "2026年3月销售业绩数据报告"`

### Step-back Prompting（后退提问）
将具体问题抽象为更通用的问题，检索到更广泛的相关内容。
`"GPT-4 的上下文窗口是多少" → "GPT-4 的技术规格和能力是什么"`

### Multi-Query（多查询扩展）
将一个问题拆分为多个子查询并行检索，提高覆盖面。
`"RAG 和微调哪个好" → ["RAG 的优势", "微调的优势", "RAG vs 微调对比"]`
可以构成 [[2Multi-Query RAG]]
### HyDE（假设性文档嵌入）
先让 LLM 生成一个「假设性答案」，用它做 Dense 检索，因为答案和文档在向量空间更接近。

### Query Routing（查询路由）
根据问题类型将查询路由到不同的检索源（向量库、SQL 数据库、API 等）。

## 与其他模块的关系
- **输出** → 传递给 Retrieval 模块进行检索
- **依赖** ← 在 Conversational RAG 中依赖 Memory 模块提供历史上下文
