从理论到落地，工程实践是 RAG 系统成败的关键。本节覆盖主流流程、推荐组合和常见陷阱。
## 主流综合流程（11 步）
1. **数据接入** — `PDF / 网页 / 表格 / API / 数据库 / 图片`
2. **预处理** — `清洗 → 去重 → 结构解析 → 元数据抽取`
3. **切块** — `语义切块 / parent-child / hierarchical`
4. **建索引** — `BM25 + Vector DB + Metadata + 可选 Graph`
5. **Query 处理** — `rewrite / step-back / multi-query / routing`
6. **召回** — `hybrid retrieve topN`
7. **精排** — `reranker → topK`
8. **压缩** — `去重 / 摘要 / 证据提纯`
9. **生成** — `带引用 answer synthesis`
10. **校验** — `grounding / consistency / refusal / guardrail`
11. **反馈闭环** — `日志 → 评测 → 调参 → 重建索引`

---
## 工程默认组合推荐
### 基础版
`Contextual Chunking + BM25 + Dense + RRF + Reranker + Citation`
适用：一般文档问答，快速上线

### 进阶版
`Rewrite + Multi-Query + Hybrid + Reranker + Compression + Verification`
适用：需要高质量回答的产品级系统

### 复杂知识库版
`Hierarchical + GraphRAG/LightRAG + Agentic Retrieval + Citation`
适用：大规模企业知识库、跨文档关联

### 高风险场景版
`Strict Retrieval + Strong Reranker + Evidence-only Generation + Refusal`
适用：金融、法律、医疗等零容错场景

## ⚠️ 常见失败点

<aside> 🚫

这些是最常见的 RAG 系统失败原因，务必逐项检查：

</aside>
1. **切块过碎/过大** — 过碎丢失上下文，过大引入噪声。建议按语义边界切块并使用 parent-child 策略
2. **只做 Dense 不做 Sparse** — Dense 擅长语义匹配但弱于精确关键词，必须搭配 BM25
3. **只召回不精排** — 召回 top-100 直接喂模型，噪声过多。加入 Reranker 筛选 top-5~20
4. **上下文堆太多不压缩** — 塞满上下文窗口反而降低生成质量。需做去重、摘要、提纯
5. **无引用、无拒答** — 无法验证回答来源，无法在信息不足时拒绝回答
6. **无元数据过滤** — 不区分时间、权限、文档类型，检索结果杂乱
7. **无评测闭环** — 没有监控召回率和忠实性，无法持续优化
