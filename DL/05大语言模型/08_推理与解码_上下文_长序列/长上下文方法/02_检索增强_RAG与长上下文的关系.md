# 检索增强生成 (RAG) 与长上下文的关系

## 1. 问题背景
当上下文长度超出模型能力时，有两种主要解决方案：
1.  **扩展上下文窗口**：通过 RoPE 扩展、稀疏 Attention 等技术增加模型的上下文容量。
2.  **检索增强生成 (RAG)**：不把所有信息塞入上下文，而是按需检索相关片段。

## 2. RAG 核心流程
### 2.1 基本架构
```
[用户 Query] --> [检索器 Retriever] --> [相关文档片段 Top-K]
                                            |
                                            v
                        [LLM 生成器 Generator] <-- [拼接: Query + Retrieved Docs]
                                            |
                                            v
                                        [回答 Answer]
```

### 2.2 关键组件
- **Embedding Model**：将 Query 和文档库编码为稠密向量。
- **向量数据库 (Vector DB)**：存储文档 Embedding，支持 ANN (近似最近邻) 检索。
- **Reranker (可选)**：对检索结果进行精排。

## 3. RAG vs 长上下文：选择哪个？
| 维度 | RAG | 长上下文 |
|------|-----|----------|
| **上下文容量** | 理论上无限（靠检索动态加载） | 受限于模型架构（如 128K） |
| **延迟** | 增加检索延迟 | 上下文越长，推理越慢 |
| **成本** | Token 数可控，推理成本低 | 长上下文 Token 数多，成本高 |
| **信息完整性** | 可能遗漏未检索到的关键信息 | 所有信息都在上下文中 |
| **适用场景** | 知识库问答、文档搜索 | 长文档摘要、多轮对话 |

## 4. 语音大模型中的 RAG (Audio RAG)

### 4.1 语音检索的特殊性
在 Audio LLM 中，RAG 的应用与文本有显著差异。

#### 文本 RAG
- 用户 Query 是文本。
- 检索单元是文本段落。

#### Audio RAG
- **Query 可能是语音**：用户提问是音频，需要先 ASR 转文本，或直接使用 Audio Embedding 检索。
- **检索单元是音频片段**：例如在播客/会议记录中检索"刚才讨论的那个话题"。

### 4.2 Audio Embedding
为了支持语音检索，需要训练能够表征语音语义的 Embedding 模型：
- **CLAP (Contrastive Language-Audio Pretraining)**：类似 CLIP，学习音频和文本的共享表示空间。
- **用途**：
  - 用文本 Query 检索相关音频（如"找到所有提到'销售额'的会议片段"）。
  - 用音频 Query 检索相似音频（如"找到和这段旋律相似的歌曲"）。

### 4.3 应用案例
#### 案例 1：播客问答
- **场景**：用户上传了 2 小时的播客音频，并提问"主持人对 AI 发展的看法是什么？"。
- **流程**：
  1.  将播客切分为若干片段（如每 30 秒一段）。
  2.  使用 Whisper 转录每个片段，存入向量数据库。
  3.  用户提问后，检索最相关的 5 个片段。
  4.  将这 5 个片段的转录文本 + 原始问题送入 Audio LLM 生成回答。
- **优势**：无需将 2 小时的音频全部编码，大幅降低推理成本。

#### 案例 2：会议助手
- **场景**：实时会议记录，用户随时可以问"刚才 Tom 说的那个数字是多少？"。
- **流程**：
  1.  实时 ASR + Speaker Diarization（说话人分离）。
  2.  每发言一段就索引入向量数据库。
  3.  用户提问时，检索相关发言片段，并定位到具体的时间戳。

### 4.4 RAG + 长上下文的结合 (Long-Context RAG)
- **策略**：先用 RAG 检索候选片段，再将 Top-K 片段完整放入长上下文（而非截断）。
- **语音优势**：音频片段的边界清晰（按时间切分），比文本更容易保持语义完整性。
