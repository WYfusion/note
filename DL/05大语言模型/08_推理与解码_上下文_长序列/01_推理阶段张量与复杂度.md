# 推理阶段：张量流动与复杂度分析

## 1. 推理的两大阶段
LLM 的推理过程通常分为两个阶段：**Prefill（预填充）** 和 **Decode（解码）**。

### 1.1 Prefill 阶段（Prompt Processing）
- **输入**：完整的 Prompt Token 序列 $X = [x_1, x_2, \dots, x_n]$。
- **计算**：并行计算所有 Token 的 Attention。
- **输出**：生成第一个 Token $x_{n+1}$，并计算出所有输入 Token 的 KV Cache。
- **特性**：计算密集型（Compute-bound），能够充分利用 GPU 并行计算能力。

### 1.2 Decode 阶段（Token Generation）
- **输入**：上一个生成的 Token $x_{t}$。
- **计算**：自回归生成，每次只处理一个 Token，利用之前缓存的 KV Cache。
- **输出**：生成下一个 Token $x_{t+1}$。
- **特性**：访存密集型（Memory-bound），受限于显存带宽（Memory Bandwidth），因为每次都需要读取庞大的 KV Cache 但只进行少量的矩阵乘法。

## 2. 复杂度分析
假设模型参数量为 $P$，层数为 $L$，隐藏层维度为 $h$，序列长度为 $S$。

### 2.1 时间复杂度
- **Attention 计算**：$O(S^2 \cdot h)$。
- **Linear 层计算**：$O(S \cdot h^2)$。
- **总复杂度**：随着序列长度 $S$ 增长，Attention 的计算量呈二次方增长。

### 2.2 空间复杂度（显存占用）
显存主要由两部分组成：
1.  **模型权重（Model Weights）**：固定占用，例如 7B 模型 FP16 约占 14GB。
2.  **KV Cache**：动态增长，与序列长度 $S$ 和 Batch Size $B$ 成正比。
    $$ \text{KV Cache Size} = 2 \times L \times B \times S \times h \times \text{Precision} $$
    - $2$ 代表 Key 和 Value。

## 3. 语音大模型（Audio LLM）的特殊性

### 3.1 序列长度爆炸
语音信号的信息密度低于文本，导致序列长度极长。
- **文本**：1秒语音 $\approx$ 2-3 个单词 $\approx$ 3-4 个 Token。
- **语音**：
  - **Raw Waveform**：16kHz 采样率 $\rightarrow$ 16,000 个点/秒。
  - **Mel Spectrogram**：10ms 帧移 $\rightarrow$ 100 帧/秒。
  - **Audio Token (Codec)**：如 EnCodec/SoundStream，通常压缩到 25-75 Hz（即 25-75 Token/秒）。

**结论**：处理 1 分钟的音频，Audio LLM 需要处理 $60 \times 50 = 3000$ 个 Token，而对应的文本可能只有 150 个 Token。这使得 Audio LLM 在推理时面临更严峻的 **KV Cache 显存压力** 和 **Attention 计算开销**。

### 3.2 编码器-解码器差异
- **Encoder-Decoder 架构**（如 Whisper）：
  - Encoder 处理音频（并行，Prefill 阶段），计算量大但只运行一次。
  - Decoder 生成文本（自回归，Decode 阶段），序列较短。
- **Decoder-only 架构**（如 AudioGPT, VALL-E）：
  - 音频和文本都作为 Token 序列，自回归生成音频时，序列极长，Decode 阶段的延迟显著。
