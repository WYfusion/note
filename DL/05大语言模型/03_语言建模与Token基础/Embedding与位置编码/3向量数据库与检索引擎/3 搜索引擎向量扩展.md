# 搜索引擎向量扩展

> [!abstract] 模块定位
> 已有搜索引擎栈时，最自然的路线通常不是再上一套独立向量数据库，而是把向量召回直接并入全文检索链路，实现 lexical retrieval、filter、aggregation、rerank 与 semantic retrieval 的统一。

## 什么时候应优先选择搜索引擎路线

> [!note] 这条路线的核心价值
> 搜索引擎扩展追求的不是单一向量性能极限，而是把 BM25、字段过滤、聚合、规则排序、向量检索放进同一查询平面。

| 产品 | 典型优势 | 更适合什么团队 |
| --- | --- | --- |
| **Elasticsearch** | dense vector、kNN、成熟生态 | 已有 ES 技术栈的团队 |
| **OpenSearch** | AWS 生态、k-NN 集成 | 以云生态为主的搜索系统 |
| **Vespa** | 多阶段排序、tensor 表达力强 | 对 ranking / rerank 要求高的团队 |

## 先理解它和专用向量数据库的差别

| 维度 | 搜索引擎扩展 | [[2 专用向量数据库实战|专用向量数据库]] |
| --- | --- | --- |
| 向量检索 | 通常够用，但不是唯一目标 | 向量优先，能力更完整 |
| BM25 / 全文检索 | 原生强项 | 往往要额外组合 |
| Hybrid 检索 | 查询层天然一体化 | 常需自己设计融合策略 |
| 运维路径 | 已有搜索集群时迁移平滑 | 需要新增检索系统 |
| 最适合 | 内容搜索、站内搜索、商品搜索 | 独立召回平台、海量向量检索 |

> [!quote] Elastic 的官方边界
> Elasticsearch 明确区分 approximate kNN 与 exact kNN：前者适合大多数生产工作负载，后者适合小数据集或对精确评分要求更高的场景。
> 参考：[Elasticsearch kNN Search](https://www.elastic.co/docs/solutions/search/vector/knn)

## Elasticsearch：最小 hybrid retrieval 示例

```python
from elasticsearch import Elasticsearch
import numpy as np

es = Elasticsearch("http://localhost:9200")

es.indices.create(
    index="docs",
    mappings={
        "properties": {
            "title": {"type": "text"},
            "content": {"type": "text"},
            "tenant_id": {"type": "keyword"},
            "category": {"type": "keyword"},
            "embedding": {
                "type": "dense_vector",
                "dims": 768,
                "index": True,
                "similarity": "cosine",
            },
        }
    },
    ignore=400,
)

query_vector = np.random.randn(768).tolist()

results = es.search(
    index="docs",
    body={
        "query": {
            "bool": {
                "must": [{"match": {"content": "向量检索 与 PostgreSQL"}}],
                "filter": [
                    {"term": {"tenant_id": "tenant_7"}},
                    {"term": {"category": "tech"}},
                ],
            }
        },
        "knn": {
            "field": "embedding",
            "query_vector": query_vector,
            "k": 50,
            "num_candidates": 200,
            "boost": 0.6,
        },
        "size": 10,
    },
)

for hit in results["hits"]["hits"]:
    print(hit["_score"], hit["_source"]["title"])
```

## Vespa：适合复杂排序链路

```python
vespa_schema = """
schema doc {
    document doc {
        field title type string {
            indexing: summary | index
        }
        field content type string {
            indexing: summary | index
        }
        field embedding type tensor<float>(d[768]) {
            indexing: attribute | index
            attribute {
                distance-metric: angular
            }
            index {
                hnsw {
                    max-links-per-node: 16
                    neighbors-to-explore-at-insert: 200
                }
            }
        }
    }

    rank-profile hybrid {
        inputs {
            query(q_embedding) tensor<float>(d[768])
        }
        first-phase {
            expression: bm25(title) + bm25(content) + closeness(field, embedding)
        }
    }
}
"""
```

> [!tip] 什么时候 Vespa 特别有优势
> 当你的问题不只是“找近邻”，而是“先召回，再多阶段排序，再结合业务特征重排”时，Vespa 这类搜索系统会比纯向量数据库更自然。

## 与 PostgreSQL 路线的边界

| 路线 | 优势 | 局限 |
| --- | --- | --- |
| [[4 pgvector 关系数据库向量扩展|PostgreSQL + pgvector]] | SQL、事务、`JOIN`、全文检索一体化 | 搜索排序与检索平台化能力较弱 |
| 搜索引擎扩展 | 全文检索、聚合、向量、排序统一 | 数据一致性和事务边界通常不如关系库强 |
