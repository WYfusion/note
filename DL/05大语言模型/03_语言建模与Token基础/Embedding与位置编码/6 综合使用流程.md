一个完整的向量检索系统包含**离线构建**、**在线检索**和**持续优化**三个阶段。

---

## 6.1 离线构建

1. **数据采集**：收集文档、网页、代码、多媒体等原始数据
2. **清洗与去重**：去噪、去重、标准化格式
3. **结构切分**：按标题/段落/表格/代码块/页面切块
4. **上下文补全**（可选）：Contextual Retrieval / Late Chunking
5. **生成 Embeddings**：可同时生成 dense + sparse 向量
6. **写入向量数据库**：批量 upsert
7. **建索引**：HNSW / IVF / PQ 等
8. **建 metadata 索引**：关键词、标签、权限、时间等

---

## 6.2 在线检索

1. **用户 Query** → query embedding / sparse query
2. **多路召回**：
    - Dense recall（向量相似度）
    - Sparse/BM25 recall（关键词匹配）
    - Filter recall（元数据筛选）
    - 可选：graph / rule recall
3. **融合去重**：合并多路结果，去除重复
4. **Rerank**：用 Cross-Encoder 精排
5. **取 top-n 证据** → 交给 LLM 生成答案
6. **返回**：答案 + 引用 + 证据片段

---

## 6.3 持续优化

- 调 chunk size / overlap
- 调 embedding 模型（升级或微调）
- 调索引参数（HNSW 的 M / ef、IVF 的 lists / probes、PQ 位数）
- 调 hybrid 权重（dense vs sparse 的比例）
- 调 rerank top-k
- Hard negative 挖掘
- 离线 benchmark + 在线 A/B 测试

---

<aside>
⚠️

**工程常见误区**：只做单向量召回，不做 hybrid / rerank / 结构切分。这会导致检索质量远低于预期，尤其在专业领域和长文档场景。

</aside>

---

<aside>
🔑

**检索质量 ≠ 生成质量**：好的检索是好的生成的前提，但两者需要分开评测。检索评测看 Recall/MRR，生成评测看答案准确性/忠实度。

</aside>
