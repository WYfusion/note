不同场景需要不同的嵌入模型 + 向量数据库 + 检索策略组合。以下是 5 种最实用的方案和选型速记。

---

## 5.1 基础通用型（默认推荐）

- **嵌入模型**：OpenAI / Voyage / E5 / BGE
- **向量数据库**：Qdrant / Pinecone / pgvector / Elasticsearch
- **检索流程**：dense + metadata filter + rerank
- **适用**：通用搜索、问答、推荐

---

## 5.2 企业知识库 / RAG 主流型

- **召回**：Hybrid（dense + BM25/SPLADE）
- **精排**：Cross-Encoder / Reranker
- **生成**：LLM with citations
- **特点**：当前最稳、最通用的 RAG 方案

---

## 5.3 长文档 / PDF / 代码库

- 长上下文 embeddings（Jina v3/v4、Nomic）
- Late Chunking / Contextual Retrieval
- 文档结构化切块（标题、表格、代码块、图注）
- rerank 必加

---

## 5.4 高精度检索

- 多向量（ColBERT / BGE-M3 / Jina-ColBERT）
- 多阶段排序
- 代价：索引更大、实现更复杂、吞吐更低
- 回报：显著更高的检索质量

---

## 5.5 多模态知识库

- text + image / PDF embeddings
- hybrid + multimodal rerank
- 适合手册、论文、报表、扫描件、图表库

---

## 选型速记

<aside>
⚡

**最快落地** → pgvector / Chroma / Qdrant + 单向量 dense

</aside>

<aside>
✅

**要稳要效果** → Hybrid + Rerank，不要只做纯向量 top-k

</aside>

<aside>
📄

**长文高质量** → 长上下文 embedding + Late Chunking + 结构化切块

</aside>

<aside>
🏆

**顶级检索质量** → multi-vector / ColBERT 路线（代价：索引大、吞吐差）

</aside>

<aside>
🔗

**与现有业务库绑定** → PostgreSQL + pgvector 或 Elasticsearch + vector，不一定需要独立向量数据库

</aside>
