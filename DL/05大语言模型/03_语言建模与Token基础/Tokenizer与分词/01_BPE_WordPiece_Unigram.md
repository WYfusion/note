# BPE、WordPiece、Unigram 三大算法详解

> [!abstract] 摘要
> 子词分词是 LLM Tokenization 的核心技术。本节详细对比 BPE、WordPiece 和 Unigram 三大算法的原理、训练过程、推理逻辑和优缺点，帮助理解不同分词器的选择依据。

## 0. 统一概念：子词分词在做什么？

给定一句话，分词器要把它变成一串 token（再映射成 id）。

> [!important] 子词分词的核心思路
> **子词分词（Subword Tokenization）**的思路是：词表里既保留常见词，也保留常见「片段」，让任意新词都能拆成若干片段来表示，从而减少 `[UNK]`。

### 分词流程

```
原始文本: "I love natural language processing"
        ↓
    初始符号化: ["I", "love", "natural", "language", "processing"]
        ↓
    子词分割: ["I", "love", "na", "tural", "language", "process", "ing"]
        ↓
    Token ID 映射: [101, 2345, 1024, ...]
```

## 1. BPE（Byte-Pair Encoding）字节对编码

### 1.1 核心思想

> [!important] 从小到大造积木
> 经典的子词 BPE（NMT 里广泛使用的那种）通常从**字符级**开始，不断把语料里最常一起出现的相邻符号对合并成新符号；重复直到达到目标（比如合并次数或词表大小）。这套「合并历史」就是 BPE 的核心模型。

> [!tip] 直观理解
> 你可以把它理解为：**BPE = 「从小到大造积木」，每一步都把最常见的相邻积木粘起来，最终得到一套子词词表（以及合并规则）。**

**历史背景**：
- Sennrich 等人在 NMT 中推广了用 BPE 把稀有词拆成子词，从而缓解 OOV 问题，这是子词 BPE 在 NLP 里大规模使用的重要来源。

### 1.2 BPE 训练阶段学到的「模型」是什么？

经典的子词 BPE（NMT 里广泛使用的那种）从**字符级**开始，不断把语料里最常一起出现的相邻符号对合并成新符号；重复直到达到目标（比如合并次数或词表大小）。

**合并操作示例**：
```
初始: "h u g"
      ↓ (找到最频繁对 "u"+"g", 合并为 "ug")
合并: "h ug"
      ↓ (继续找最频繁对, 合并 "h"+"ug" 为 "hug")
合并: "hug"
...
最终: "hug" → 添加到词表
```

### 1.3 BPE 推理（真正分词）怎么切？

当你拿到新文本时，BPE 通常会：
1. 先把词切成初始符号序列（常见是字符；也有 byte-level 变体是字节）
2. 按训练时学到的「合并规则」依次应用，把能合并的地方合并掉
3. 最终得到 token 序列（确定性的）

> [!warning] 确定性
> 因此 **BPE 的分词结果基本是确定的**：同一个词、同一套 merges，切出来就固定。

### 1.4 BPE 的优缺点（直观版）

| 特性 | BPE | 说明 |
|------|----|------|
| **优点** | ✅ 简单、快 | 工业界非常成熟（很多 GPT 系列历史上也使用 BPE 变体） |
|  | ✅ 通常能把绝大多数新词拆开 | 减少 `[UNK]` |
| **缺点/容易踩坑点** | ❌ 分词是「按合并规则执行」的 | 缺少 Unigram 那种显式概率最优化 |
|  | 同一个词通常只有一种切法（确定性），多样性较弱 |
|  | 许多 BPE 实现依赖「先按空格切词」的预切分 | SentencePiece 这种工具就是为了解决「直接从原始句子训练」的工程痛点（见后文） |

## 2. WordPiece 词块

### 2.1 WordPiece 在 BERT 系列里是什么地位？

WordPiece 是 Google 系列（尤其 BERT）最典型的子词方案之一；BERT 的官方实现和文档中明确提到使用 WordPiece，并展示了带 `##` 的子词形式。

### 2.2 WordPiece 的核心「分词算法」是什么？

**推理时（分词时）**，WordPiece 的最核心特征是：
- **只保存最终词表（vocab）**（不一定保存/暴露 BPE 那种 merges 列表）
- **对每个词使用**贪心的 longest-match-first**：从当前位置起，优先匹配词表中最长的子串；匹配到后继续处理剩余部分
- **对「非词首的子词」加一个前缀**（BERT 常见是 `##`），表示这是一个词的中间部分

**分词示例**：
```
输入: "unhappiness"
词表: ["un", "unhappi", "ness", "##ness"]

步骤1: 找最长匹配 → "un" (在词表中)
步骤2: 处理剩余 → "happiness"
步骤3: 对 "happiness" 找最长匹配 → "happi" (在词表中)
步骤4: 处理剩余 → "ness" (在词表中)
最终: ["un", "##happi", "##ness"]
```

Hugging Face 课程里专门对比了 WordPiece vs BPE：WordPiece 分词是「按词表做最长匹配切分」，而不是「按 merges 顺序合并」，所以同一个词在两者下可能切出不同结果。

### 2.3 WordPiece 的 `[UNK]`：什么时候会出现？

WordPiece 通常有显式的 `unk_token`（例如 `[UNK]`）。

> [!tip] 跨语言场景
> 当一个「词」无法被词表切分表示（或超过某些限制，比如单词太长）时，会直接回退成 `[UNK]`。

> [!note] 中文场景
> 这也解释了：用 `bert-base-chinese` 去切英文时，很多词会变 `[UNK]` ——因为中文 BERT 的词表对英文覆盖不够。

### 2.4 WordPiece 的优缺点（直观版）

| 特性 | WordPiece | 说明 |
|------|----------|------|
| **优点** | ✅ 与 BERT 的生态强绑定 | 现成 vocab、现成预训练模型、工具链完善 |
|  | ✅ 分词实现直观 | 按词表做最长匹配；且 `##` 让「是否是词中间子词」一眼可见 |
| **缺点/注意点** | ❌ 对词表覆盖依赖更强 | 覆盖差就更容易 `[UNK]`（尤其跨语言/跨域） |
|  | ❌ 分词同样是确定性的 | 不像 Unigram 那样天然支持「多种切法 + 采样」作为正则 |

## 3. Unigram（Unigram Language Model Tokenization）

### 3.1 Unigram 训练阶段学到的「模型」是什么？

Unigram 跟 BPE/WordPiece 最大的结构性差异是：
- **它不是「合并规则」驱动**（不靠 merges）
- 它维护的是一个「子词词表 + 每个子词的概率（或打分）」，然后用一个训练目标（常见是对数似然/损失）来决定保留哪些子词。

**训练流程**：
1. 在当前词表与 unigram LM 下计算训练数据的损失（对数似然形式）
2. 评估「移除某个 token 会让总损失增加多少」
3. 每轮移除一部分（常见 10% 或 20%）对损失影响最小的 token
4. 重复直到词表达到目标大小

> [!tip] 直观理解
> 你可以把它理解为：**Unigram = 「先准备很多候选子词，然后用概率模型 + 剪枝把不重要的子词删掉」，最后得到一个概率化的子词词表。**

### 3.2 Unigram 推理（真正分词）怎么切？

因为 Unigram 不依赖 merges，所以对一个词/句子，往往存在多种切分方式。Unigram 会用词表中各 token 的概率，去选一条整体概率最大的切分（实践中常用 Viterbi / DP 来找最优路径）。

Hugging Face 的说明里举了「同一个词可能有多种切法」的例子，并指出 Unigram 最终会选择最可能的切法，也可以按概率做采样。

### 3.3 为什么大家一提 Unigram 就会提 SentencePiece？

因为 **SentencePiece** 这个工具/库把 Unigram（以及 BPE）做成了非常工程化、语言无关的训练与分词方案：
- 可以直接从「原始句子」训练，不强依赖预切分
- 把空格当作普通符号处理（常见用 `▁` 显式表示空格），从而做到更「可逆/无歧义」的处理
- 支持 unigram 与 BPE 两种模型类型，并支持采样（subword regularization）

### 3.4 Unigram 的优缺点（直观版）

| 特性 | Unigram | 说明 |
|------|--------|------|
| **优点** | ✅ 天然概率化 | 同一个词允许多种切法 |
|  | ✅ 训练时可用采样做正则（subword regularization） | 在一些任务上更鲁棒 |
|  | ✅ 结合 SentencePiece 时 | 对多语言、无空格语言、更「端到端」的管线友好 |
| **缺点** | ❌ 推理时要在多种切法里找最优 | 实现和理解上比「应用合并规则/最长匹配」更复杂一些 |

## 4. 三者对比表（建议你用这张表形成「脑内索引」）

| 维度 | BPE | WordPiece | Unigram |
|------|----|----|------|
| **核心思想** | 频繁相邻对合并（从小到大造 token） | 词表 + 贪心最长匹配切分，常用 `##` 标记词中子词 | 概率化词表；用 unigram LM 选最可能切分 |
| **「模型文件」更像什么** | merges + vocab（主要是 vocab，WordPiece 分词不按 merges） | vocab + token 概率（以及用于最优切分的机制） |
| **推理是否确定** | 基本确定（贪心） | 可确定（取最优）也可采样（正则） |
| **常见搭配** | 许多 BPE/byte-level BPE 系列模型 | BERT 系列典型 | SentencePiece 的 unigram 模型（很多多语言/无空格语言场景喜欢） |
| **采样支持** | ❌ 无 | ❌ 无 | ✅ 有（subword regularization） |

---

## 5. 在 Hugging Face / transformers 里，你怎么「对应到它们」？

你在 `transformers` 里常见的现象是：

- **BERT 家族**：通常是 WordPiece（并且 fast tokenizer 支持你做 `word_ids()`、offset 对齐等）
- **SentencePiece 系列（例如很多 T5、LLaMA 类模型的 tokenizer 实现）**：经常是 SentencePiece（背后可能是 unigram 或 BPE），并且空格会以 `▁` 这类方式显式编码。
- **你想「确认某模型到底是哪种」**：最靠谱的是看 `tokenizer` 的类名/配置以及它的 `tokenizer.json` / `vocab` 文件结构，而不是只看模型名字。
