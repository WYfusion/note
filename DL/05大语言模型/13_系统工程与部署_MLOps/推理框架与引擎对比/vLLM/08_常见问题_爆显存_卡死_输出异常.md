# 常见问题：爆显存、卡死与输出异常

## 1. 爆显存 (OOM)

### 1.1 现象
服务启动失败，或者运行中途崩溃，报错 `CUDA out of memory`。

### 1.2 解决方案
1.  **调低 `gpu_memory_utilization`**: 默认 0.9 太激进，改为 0.8 或 0.85。
2.  **限制 `max_model_len`**: 某些模型配置文件里的上下文长度是 32k 甚至 128k，vLLM 会尝试预留相应的 KV Cache。手动限制为 4096 或 8192。
    ```bash
    --max-model-len 4096
    ```
3.  **检查 Input Length**: 对于 Audio LLM，一段 5 分钟的音频可能产生数万个 Token，直接撑爆显存。需在应用层进行切片。

---

## 2. 请求卡死 (Stuck / Timeout)

### 2.1 NCCL Timeout
多卡推理时，如果进程间通信受阻，会导致卡死。
*   **解决**: 确保 Docker 启动时加了 `--ipc=host`。

### 2.2 调度死锁
如果 `max_num_seqs` 设置过大，而显存不足以容纳哪怕一个请求的 KV Cache，vLLM 可能会陷入死循环（不断 Swap out 又 Swap in）。
*   **解决**: 减小 `max_num_seqs`。

---

## 3. 输出异常 (Gibberish / Repetition)

### 3.1 停不下来 (Never Ending)
模型一直生成重复内容，直到达到 `max_tokens`。
*   **原因**: EOS Token ID 配置错误，或者模型没有正确预测 EOS。
*   **解决**: 显式指定 `stop_token_ids`。

### 3.2 乱码
*   **原因**: 采样参数 `temperature` 过高，或者量化模型精度损失严重。
*   **Audio 特有**: 如果输入的音频特征预处理不当（如采样率不对），模型会输出无意义的文本。务必确保输入音频是 16kHz（或其他模型要求的采样率）。
