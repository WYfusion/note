# 吞吐与延迟：并发与批处理调参

在生产环境中，我们需要根据业务需求（是追求低延迟的实时对话，还是高吞吐的离线转录）来调整 vLLM 的参数。

## 1. 核心调优参数

### 1.1 `max_num_seqs` (最大并发数)
*   **定义**: vLLM 可以在同一时刻处理的最大请求序列数。
*   **默认值**: 256。
*   **调优**:
    *   **高吞吐场景**: 调大此值（如 512 或 1024），让 GPU 尽可能满载。
    *   **低延迟场景**: 保持默认或适当调小，避免排队等待时间过长。

### 1.2 `max_num_batched_tokens`
*   **定义**: 一个 Batch 中包含的最大 Token 总数（Prompt + Generated）。
*   **作用**: 防止某个 Batch 过大导致 OOM。
*   **Audio LLM 注意事项**: 音频特征通常会被映射为大量的 Token（例如 30秒音频可能对应 1500 个 Token）。这意味着处理 Audio 请求时，这个限制更容易被触发。

---

## 2. 性能监控

vLLM 启动时会打印性能统计信息，或者通过 `/metrics` 接口暴露 Prometheus 指标。

*   **Avg prompt throughput**: 每秒处理的 Prompt Token 数。
*   **Avg generation throughput**: 每秒生成的 Token 数。
*   **GPU KV cache usage**: KV Cache 的使用率。如果长期接近 100%，说明显存不足，请求被频繁换出（Swapped out）到 CPU，严重影响性能。

---

## 3. 针对 Audio LLM 的建议

### 3.1 预估 Token 用量
对于 Qwen-Audio，一段音频输入会被转换为特定数量的 Embedding 向量。vLLM 内部会将这些向量视为 Token。
*   **计算**: 假设 1秒音频 = 50 Tokens。
*   **影响**: 如果用户上传了 1分钟的音频（3000 Tokens），这相当于一个很长的 Prompt。
*   **策略**: 适当调低 `max_num_seqs`，因为每个请求占用的显存（KV Cache）和计算量都比纯文本请求大得多。

### 3.2 异步引擎 (AsyncLLMEngine)
在 Python 代码中使用 `AsyncLLMEngine` 可以获得比 HTTP Server 更细粒度的控制，适合构建复杂的语音 Agent 流水线。
