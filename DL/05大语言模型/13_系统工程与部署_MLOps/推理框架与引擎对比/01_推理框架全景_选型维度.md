# 推理框架全景与选型

选择合适的推理框架决定了系统的吞吐量和延迟。对于语音大模型（Audio LLM），由于涉及音频编码器（Encoder）和长序列生成（Decoder），选型逻辑比纯文本模型更为复杂。

## 1. 框架概览

| 框架 | 核心技术 | 适用场景 | 语音模型支持度 | 优势 | 劣势 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **HuggingFace** | PyTorch 原生 | 研究、原型验证 | ⭐⭐⭐⭐⭐ (原生支持所有) | 代码简单，生态最丰富，调试方便 | 性能较差，显存占用高，无高级 Batching |
| **vLLM** | PagedAttention | 高并发服务端 | ⭐⭐⭐⭐ (Qwen-Audio, Whisper) | **吞吐量极高**，显存管理优秀，易于集成 | 对新模型架构支持有滞后，量化支持有限 |
| **TensorRT-LLM** | CUDA Kernels | 极致性能 (NVIDIA) | ⭐⭐⭐ (需构建) | **延迟最低**，支持 FP8/INT4，算子融合最强 | 构建流程复杂，环境依赖严格，灵活性差 |
| **CTranslate2** | C++ Transformer | 生产环境 (CPU/GPU) | ⭐⭐⭐⭐ (Whisper 首选) | 轻量级，支持 INT8，CPU 性能极佳 | 主要针对标准 Transformer，对新架构支持慢 |
| **ONNX Runtime** | Graph Optimization | 端侧/跨平台 | ⭐⭐⭐⭐ | 兼容性好，适合部署 Encoder 部分 | 动态 Shape 支持不如 TRT，大模型优化不如 vLLM |

---

## 2. 语音领域的特殊选型逻辑

### 2.1 纯 ASR 任务 (如 Whisper)
*   **首选**: **Faster-Whisper (CTranslate2)** 或 **TensorRT-LLM**。
*   **理由**: Whisper 是典型的 Encoder-Decoder 结构。
    *   CTranslate2 对此类结构优化极佳，且支持 CPU/GPU 无缝切换。
    *   TensorRT-LLM 提供了官方的 Whisper 示例，性能是 PyTorch 的 5-10 倍。

### 2.2 语音理解/对话任务 (如 Qwen-Audio, Salmonn)
*   **首选**: **vLLM**。
*   **理由**: 这类模型本质上是 LLM 接一个 Audio Encoder。
    *   vLLM 的 **PagedAttention** 能完美解决长音频对话带来的 KV Cache 碎片化问题。
    *   vLLM 的 **Continuous Batching** 能处理不同长度的音频输入，避免 Padding 浪费。

### 2.3 移动端/边缘计算
*   **首选**: **ONNX Runtime** 或 **MNN/NCNN**。
*   **理由**: 资源受限，无法运行 CUDA。
    *   通常将 Audio Encoder 导出为 ONNX，LLM 部分使用 llama.cpp 或 MLC-LLM。

## 3. 决策树

1.  **是否需要极致并发 (QPS > 100)?**
    *   是 -> **vLLM** (易用) 或 **TensorRT-LLM** (极致)。
    *   否 -> 继续。

2.  **是否是标准 Whisper 模型?**
    *   是 -> **Faster-Whisper** (CTranslate2)。
    *   否 -> 继续。

3.  **是否需要运行在非 NVIDIA 硬件?**
    *   是 -> **ONNX Runtime** / **OpenVINO** (Intel) / **llama.cpp** (Mac/CPU)。
    *   否 -> **HuggingFace Transformers** (开发阶段)。
