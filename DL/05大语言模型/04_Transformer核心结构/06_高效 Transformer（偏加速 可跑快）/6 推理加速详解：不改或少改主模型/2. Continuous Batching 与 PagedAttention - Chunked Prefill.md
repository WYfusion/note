## 概述

推理加速的系统层核心：Continuous Batching（动态批处理）、PagedAttention（KV cache 分页管理）和 Chunked Prefill（分块预填充）。本页从原理到 vLLM 工程实现详述。

---

## 1. Continuous Batching

### 1.1 静态 Batching 的问题

```JavaScript
请求 A: [======长 prompt======][生成 3 token] ← 先完成
请求 B: [===短 prompt===][生成 20 token]      ← 后完成
请求 C: [====prompt====][生成 10 token]

静态 batch：必须等最慢的 B 完成，A 和 C 完成后 GPU 空闲
→ GPU 利用率低
```

### 1.2 Continuous Batching 原理

**核心**：每个 iteration（而非每个请求）重组 batch。

```JavaScript
Iteration 1: [A_decode, B_decode, C_decode]
Iteration 2: [A_decode, B_decode, C_decode]  ← A 生成完毕
Iteration 3: [D_prefill, B_decode, C_decode]  ← D 立即加入
Iteration 4: [D_decode, B_decode, C_decode]
...
```

- 请求完成后立即释放资源

- 新请求立即加入

- **GPU 几乎不空闲**

### 1.3 吞吐提升

- 相比静态 batching：**2-23x 吞吐提升**（取决于工作负载多样性）

- 原因：消除了 padding 浪费和等待浪费

## 2. PagedAttention (vLLM)

### 2.1 KV Cache 管理问题

- 每个请求需要存储 KV cache

- KV cache 大小 = $2 \times n_{layers} \times n_{heads} \times \text{seq\_len} \times d_{head}$

- 传统方案：预分配连续内存 → **内存碎片严重**

### 2.2 PagedAttention 原理

**类比操作系统的虚拟内存**：

- 将 KV cache 切成固定大小的 **page**（如 16 token/page）

- 物理内存不需要连续

- 通过 **page table** 映射逻辑位置到物理位置

```JavaScript
逻辑 KV cache:  [Page0][Page1][Page2][Page3]
                  ↓       ↓       ↓       ↓
物理内存:      [P2][..][P0][..][P3][P1]
               (不连续分配)
```

### 2.3 优势

- **内存利用率接近 100%**：碎片 < 4%（vs 传统方案 60-80%）

- **支持 copy-on-write**：beam search 时共享 KV cache

- **灵活回收**：请求结束后释放 page，立即可用

### 2.4 Attention Kernel 适配

## 3. Chunked Prefill

### 3.1 Prefill vs Decode 的冲突

- **Prefill**：处理输入 prompt，compute-bound，一次性大矩阵计算

- **Decode**：逐 token 生成，memory-bound，每步很小的计算

- 长 prompt 的 prefill 会阻塞其他请求的 decode → **TTFT（Time to First Token）不稳定**

### 3.2 Chunked Prefill 原理

将大的 prefill 请求切成小块，与 decode 请求混合调度：

```JavaScript
传统调度:
[======Prefill A (4096 tokens)======][Decode B][Decode C]
↑ B, C 必须等待 A 完成 prefill

Chunked Prefill:
[Prefill A chunk1 | Decode B | Decode C]
[Prefill A chunk2 | Decode B | Decode C]
[Prefill A chunk3 | Decode B | Decode C | Decode D]
↑ B, C 的 decode 不被阻塞
```

### 3.3 vLLM V1 的实现

- **默认开启** chunked prefill

- 调度策略：
    
    1. 优先调度所有 pending decode 请求
    
    1. 剩余 `max_num_batched_tokens` 预算用于 prefill
    
    1. 如果 prefill 超出预算 → 自动切块
    

### 3.4 关键参数

```Python
from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-2-7b-hf",
    # chunked prefill 配置
    enable_chunked_prefill=True,        # V0 需手动开；V1 默认开
    max_num_batched_tokens=4096,        # 每步最大 token 预算
    max_num_seqs=256,                   # 最大并发序列
)
```

## 4. 完整推理优化栈

```JavaScript
┌─────────────────────────────────────────┐
│          用户请求                        │
├─────────────────────────────────────────┤
│  调度层                                  │
│  ├─ Continuous Batching                 │
│  ├─ Chunked Prefill                     │
│  └─ Priority Scheduling (decode first)  │
├─────────────────────────────────────────┤
│  内存管理层                               │
│  ├─ PagedAttention (KV cache)           │
│  ├─ Prefix Caching                      │
│  └─ Preemption (swap/recompute)         │
├─────────────────────────────────────────┤
│  计算加速层                               │
│  ├─ FlashAttention                      │
│  ├─ Speculative Decoding                │
│  └─ Tensor Parallelism                  │
├─────────────────────────────────────────┤
│  硬件层                                  │
│  └─ GPU (A100/H100)                     │
└─────────────────────────────────────────┘
```

## 5. 性能基准

|优化技术|主要收益|典型提升|
|---|---|---|
|Continuous Batching|吞吐量|2-23x vs static|
|PagedAttention|KV cache 利用率|2-4x 并发请求|
|Chunked Prefill|TTFT 稳定性|P99 延迟降低 50%+|
|Prefix Caching|重复 prompt 加速|共享前缀 ~0 cost|
|Speculative Decoding|单请求延迟|2-5x decode 加速|

---

> [!important]
> 
> **推理优化不是选其一，而是全栈组合**。典型生产部署：vLLM/SGLang + FlashAttention + Continuous Batching + PagedAttention + Chunked Prefill + (可选) Speculative Decoding。
