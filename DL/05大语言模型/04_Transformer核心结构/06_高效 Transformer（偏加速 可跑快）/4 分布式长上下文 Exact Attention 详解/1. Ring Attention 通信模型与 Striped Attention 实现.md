## 概述

Ring Attention 将超长序列的 exact attention 分布到多个设备，通过 ring 通信拓扑实现通信-计算重叠。Striped Attention 解决 causal 场景下的负载不均问题。

---

## 1. Ring Attention 详解

### 1.1 问题设定

- 序列长度 $N$ 太长，单卡放不下

- 有 $P$ 个设备（GPU），每卡持有 $N/P$ 个 token

- 目标：**计算 exact attention，不做近似**

### 1.2 核心算法

**Step 1：序列分片**

- 将 $Q, K, V$ 各自均匀切成 $P$ 块

- 设备 $p$ 持有 $Q_p, K_p, V_p$

**Step 2：Ring 通信**

- 设备按环形排列：$0 \to 1 \to 2 \to \dots \to P-1 \to 0$

- 每个通信步（round）中，每个设备：
    
    1. **发送**当前的 $K, V$ 块到下一个设备
    
    1. **接收**前一个设备的 $K, V$ 块
    
    1. **计算**当前 $Q$ 块与接收到的 $K, V$ 块的 attention
    

**Step 3：Online Softmax 累积**

- 与 FlashAttention 相同的 online softmax 机制

- 每收到一个新的 $K, V$ 块就增量更新输出

### 1.3 伪代码

```Python
def ring_attention(Q_local, K_local, V_local, rank, world_size):
    """
    Ring Attention: 每个 GPU 持有序列的一个分片
    rank: 当前 GPU 编号
    world_size: GPU 总数 P
    """
    # 初始化 running stats (同 FlashAttention)
    m = torch.full((block_size,), float('-inf'))  # running max
    ell = torch.zeros(block_size)                  # running sum
    O = torch.zeros_like(Q_local)                  # running output
    
    K_recv, V_recv = K_local.clone(), V_local.clone()
    
    for step in range(world_size):
        # 异步发送当前 KV 到下一个设备
        send_req = isend(K_recv, V_recv, dst=(rank + 1) % world_size)
        
        # 计算当前 Q 与接收到的 KV 的 attention
        # (与 FlashAttention 的 online softmax 更新完全相同)
        S = Q_local @ K_recv.T * scale
        m_new = torch.max(m, S.max(dim=-1).values)
        alpha = torch.exp(m - m_new)
        P = torch.exp(S - m_new.unsqueeze(-1))
        
        ell_new = ell * alpha + P.sum(dim=-1)
        O = (O * (ell * alpha).unsqueeze(-1) + P @ V_recv) / ell_new.unsqueeze(-1)
        
        m = m_new
        ell = ell_new
        
        # 等待接收前一个设备的 KV
        K_recv, V_recv = recv(src=(rank - 1) % world_size)
        send_req.wait()  # 确保发送完成
    
    return O
```

### 1.4 通信-计算重叠

```JavaScript
时间线（设备 p）:

Step 0: [Compute(Q_p, K_p)] [Send K_p → p+1] [Recv K_{p-1}]
                             ↕ 重叠
Step 1: [Compute(Q_p, K_{p-1})] [Send K_{p-1} → p+1] [Recv K_{p-2}]
                                 ↕ 重叠
Step 2: ...
```

**关键**：只要 attention 计算时间 ≥ KV 块传输时间，通信几乎完全被隐藏。

### 1.5 显存分析

每卡只需存储：

- 自己的 $Q, K, V$ 块：$O(N/P \cdot d)$

- 一个接收缓冲：$O(N/P \cdot d)$

- Running stats：$O(N/P)$

总显存：$O(N \cdot d / P)$，**线性于每卡分片大小**

## 2. Striped Attention：解决 Causal 负载不均

### 2.1 问题

在 causal attention 中：

- Token $i$ 只 attend 位置 $0, 1, \dots, i$

- 三角形结构 → **前面的设备计算少，后面的多**

- Ring Attention 中设备 0（持有最早 token）大部分 step 空闲

### 2.2 Striped 分布策略

**原始 Ring**（连续分片）：

```JavaScript
设备 0: [token 0, 1, 2, 3]
设备 1: [token 4, 5, 6, 7]
设备 2: [token 8, 9, 10, 11]
设备 3: [token 12, 13, 14, 15]
```

→ 设备 0 的工作量远小于设备 3

**Striped**（条纹分片）：

```JavaScript
设备 0: [token 0, 4, 8, 12]
设备 1: [token 1, 5, 9, 13]
设备 2: [token 2, 6, 10, 14]
设备 3: [token 3, 7, 11, 15]
```

→ 每个设备持有均匀分布的 token，工作量更均衡

### 2.3 性能提升

- Causal attention 场景下，Striped 比原始 Ring 吞吐提升 **~1.5-2x**

- 原因：消除了三角形负载不均导致的 GPU 空闲

## 3. Sequence Parallelism 工程实践

### 3.1 与 Ring Attention 的区别

- **Ring Attention**：专门解决 attention 计算的跨设备分布

- **Sequence Parallelism**：更广义，将整个 Transformer 层（含 FFN、LayerNorm 等）按序列维度切分

### 3.2 Megatron-LM 的 Sequence Parallelism

- 在 Tensor Parallelism 的基础上

- 非 TP 部分（LayerNorm, Dropout）按序列维度切分

- 通过 all-gather 和 reduce-scatter 实现通信

### 3.3 实践要点

```Python
# 典型组合（长序列训练）
config = {
    'tensor_parallel': 4,
    'pipeline_parallel': 2,
    'sequence_parallel': True,      # 序列维度切分 activation
    'flash_attention': True,         # FlashAttention
    'activation_checkpointing': True, # 减少激活显存
    'ring_attention': True,           # 超长序列时开启
}
```

## 4. 工程选型

> [!important]
> 
> - **单卡能放下上下文** → FlashAttention 即可
> 
> - **单卡放不下，需要 exact** → Ring Attention
> 
> - **Causal + Ring** → 优先 Striped Attention
> 
> - **长序列训练** → Ring Attention + Sequence Parallelism + Activation Checkpointing
