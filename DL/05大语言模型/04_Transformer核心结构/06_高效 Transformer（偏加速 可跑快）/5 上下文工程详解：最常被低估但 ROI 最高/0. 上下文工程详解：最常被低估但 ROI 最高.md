## 定位

- **不改模型主干**，只减少有效输入长度 $n \to n'$

- 大部分 attention / prefill / decode 成本随之下降

- **工程 ROI 极高**，最稳、最通用、最先该做

## ROI 是什么

ROI 是 `Return on Investment`，中文通常叫**投资回报率**。它衡量的是：

> 为了获得某个收益，你投入的成本值不值得。

在最常见的形式里，ROI 可以写成：

$$
ROI = \frac{收益 - 成本}{成本}
$$

如果只看比例，也常写成：

$$
ROI = \frac{收益}{成本}
$$

这里“收益”和“成本”不是固定只指金钱，在工程里通常可以换成：

- **收益**：效果提升、时延下降、吞吐提升、显存下降、故障率降低、维护成本下降
- **成本**：开发人力、上线风险、训练/推理资源、调参时间、后续维护复杂度

### 在上下文工程里的含义

上下文工程的 ROI 高，意思不是“理论最先进”，而是：

- 它通常不需要改模型结构
- 它能直接减少输入 token 数
- 它往往可以立刻带来稳定的性能收益
- 它的实现风险比改注意力机制更低

所以在 LLM 系统里，ROI 高通常意味着：

- **少量工程投入**，带来**明显的系统收益**
- 更容易在生产环境快速落地
- 更容易复用到不同模型、不同任务、不同业务场景

## 原理

设原始输入 $n$ 个 token，压缩到 $n'$ 个：

- Attention 计算：$O(n'^2 d)$ vs $O(n^2 d)$

- Prefill 时间 ∝ $n'$

- KV cache 大小 ∝ $n'$

- 效果：**直接减** $n$ **比换注意力变体更可靠**

## 四大手段

### A. Context Selection（检索选择）

- 检索后只送最相关片段

- Top-k chunk 选择 + rerank 后拼接

- 典型流程：query → retriever → reranker → top-k chunks → LLM

- 核心：**减少无关 token 进入模型**

### B. Prompt Compression（提示压缩）

- Token / 短语 / 句子级压缩

- 删除冗余，保留关键证据

- **代表**：LLMLingua 系列
    
    - **LLMLingua**：用小模型（GPT-2/LLaMA-7B）计算 token 重要性，粗到细压缩，可达 **20x 压缩率**
    
    - **LongLLMLingua**：针对长上下文 + RAG，question-aware 压缩，解决 "lost in the middle" 问题
    
    - **LLMLingua-2**：数据蒸馏训练 BERT 级编码器做 token 分类，3-6x 快于 LLMLingua
    

### C. Hierarchical Context（分层上下文）

- 先摘要 / 先粗检索，再精读局部

- 粗到细两阶段上下文

- 减少第一轮 LLM 调用的 token 数

### D. Chunk Packing / Layout Optimization

- 降低模板冗余

- 减少重复 system/context 片段

- 多样本共享前缀时做 **前缀复用**（prefix caching）

---

## 工程判断

> [!important]
> 
> **对大多数线上系统，先减上下文 → 收益常比更换注意力变体更可靠。**
> 
> 因为它直接减少 $n$，而 $n$ 影响几乎所有下游环节：attention、KV cache、prefill、decode。
