## 概述

上下文工程的核心：不改模型，减少 token 数。本页详细介绍 LLMLingua 系列的压缩算法原理，以及 RAG 中的 Context Selection 和 Prefix Caching 实践。

---

## 1. LLMLingua：基于 Perplexity 的粗到细压缩

### 1.1 核心思想

- 用小语言模型（GPT-2-small / LLaMA-7B）评估每个 token 的 "重要性"

- 重要性度量：**条件概率**（perplexity 视角）

- 低概率 token = 高信息量 = 应保留

- 高概率 token = 可预测 = 可删除

### 1.2 三步压缩

**Step 1：Budget Controller（预算控制器）**

- 将 prompt 分为多个组件（instruction / demos / question）

- 按语义重要性分配各组件的压缩预算

- 高语义密度的部分少压缩

**Step 2：Token-level Iterative Compression**

- 对每个组件，逐 token 计算条件概率

- 保留概率最低的 token（信息量最大）

- 迭代多轮，考虑 token 间依赖

**Step 3：Distribution Alignment**

- 通过 instruction tuning 对齐小模型与目标 LLM 的分布

- 使压缩后的 prompt 对目标 LLM 更 "可读"

### 1.3 性能

- 可达 **20x 压缩率**，性能损失极小

- GSM8K 上 20x 压缩仍保持 ~95% 原始性能

## 2. LongLLMLingua：针对长上下文的 Question-Aware 压缩

### 2.1 解决的问题

- **Lost in the Middle**：LLM 对长上下文中间部分的信息关注不足

- 长 prompt 成本高且性能下降

### 2.2 核心改进

**① Question-Aware 粗粒度压缩**

- 计算每个文档/chunk 与 question 的相关性

- 先移除不相关的整个文档段

**② 动态压缩率**

- 相关性高的段落少压缩，低的多压缩

- 自适应分配 token 预算

**③ 文档重排序**

- 将最相关的文档放在开头和结尾

- 缓解 position bias

### 2.3 效果

- NaturalQuestions：**准确率提升 21.4%**，token 数减少 4x

- LooGLE benchmark：**成本降低 94%**

## 3. LLMLingua-2：数据蒸馏驱动的 Token 分类

### 3.1 架构变化

- **不再**用自回归语言模型评估 token 重要性

- 改为训练一个 **BERT 级别的 token 分类器**

- 训练数据：GPT-4 标注哪些 token 是 "必要" 的

### 3.2 方法

1. 用 GPT-4 对大量 prompt 做压缩标注

1. 训练小型 encoder（如 XLM-RoBERTa）做二分类：每个 token → keep/drop

1. 推理时一次前向即可完成压缩

### 3.3 优势

- **速度快 3-6x**：encoder 前向 vs 自回归逐 token 评估

- **任务无关**：蒸馏训练使其具备泛化能力

- **性能更强**：更好的 out-of-domain 表现

## 4. Python 使用示例

```Python
from llmlingua import PromptCompressor

# 初始化压缩器
compressor = PromptCompressor(
    model_name="microsoft/llmlingua-2-xlm-roberta-large-meetingbank",
    use_llmlingua2=True
)

# 原始 prompt
original_prompt = """
You are a helpful assistant. Given the following documents, 
answer the user's question.

Document 1: The Eiffel Tower is a wrought-iron lattice tower 
on the Champ de Mars in Paris, France. It is named after the 
engineer Gustave Eiffel, whose company designed and built the 
tower from 1887 to 1889...

Document 2: The Great Wall of China is a series of fortifications 
made of stone, brick, tamped earth, and other materials...

Document 3: ... (更多文档)

Question: When was the Eiffel Tower built?
"""

# 压缩
result = compressor.compress_prompt(
    original_prompt,
    rate=0.5,  # 压缩到 50% token
    force_tokens=["?", ".", "\n"],  # 保留标点
)

print(f"原始 token 数: {result['origin_tokens']}")
print(f"压缩后 token 数: {result['compressed_tokens']}")
print(f"压缩率: {result['ratio']:.2f}x")
print(f"压缩后 prompt:\n{result['compressed_prompt']}")
```

## 5. RAG Context Selection 实践

### 5.1 标准 RAG 流程

```JavaScript
用户 Query
  ↓
① Retriever (BM25 / 向量检索) → Top-K chunks
  ↓
② Reranker (Cross-encoder) → 重排序 Top-K
  ↓
③ 选择 Top-k' 个最相关 chunks
  ↓
④ 拼接为 context → 送入 LLM
```

### 5.2 关键优化点

**Chunk 大小选择**

- 太大：包含噪声

- 太小：丢失上下文

- 经验值：256-512 token / chunk

**Reranker**

```Python
# 使用 cross-encoder reranker
from sentence_transformers import CrossEncoder

reranker = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-12-v2')

query = "When was the Eiffel Tower built?"
chunks = ["chunk1...", "chunk2...", "chunk3..."]

scores = reranker.predict(
    [(query, chunk) for chunk in chunks]
)
# 按分数排序，取 top-k
```

## 6. Prefix Caching（前缀复用）

### 6.1 场景

- 多个请求共享相同的 system prompt / few-shot examples

- 每次都重新计算 prefill 是浪费

### 6.2 实现

- **缓存 KV cache**：对共享前缀的 KV cache 只计算一次

- **vLLM automatic prefix caching**：自动检测请求间的公共前缀

- **SGLang RadixAttention**：用 radix tree 管理 KV cache 共享

### 6.3 收益

- System prompt 长度 = 2000 token，请求量 = 1000 QPS

- 无缓存：$2000 times 1000 = 2M$ token/s 的 prefill

- 有缓存：首次 2000 token + 后续只算新增部分

- **Prefill 计算量减少 ~99.9%**（对共享前缀部分）

```Python
# vLLM 开启 prefix caching
from vllm import LLM

llm = LLM(
    model="meta-llama/Llama-2-7b-hf",
    enable_prefix_caching=True  # 自动前缀复用
)
```
