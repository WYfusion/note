# PagedAttention 如何解决 KV 内存碎片

## 问题定义

为什么长上下文在线服务里，问题不只是“KV cache 太大”，还包括碎片、页分配和请求调度？

## 直觉解释

如果把每个请求都当成一整块连续内存来管理，短请求结束、长请求追加时会造成很多空洞。PagedAttention 的思路是把 cache 拆成固定大小页面来管理，像操作系统分页一样减少碎片。

## 形式化推导

它并不改变 attention 公式，而是改变 KV cache 的存储布局：把逻辑连续的 token 历史映射到物理不必连续的 page 上，再通过页表式索引访问。

## 工程意义

这种设计的收益是：

- 更高的显存利用率。
- 更灵活的多请求调度。
- 更少的大块连续内存需求。

## 常见误解

> [!warning] 常见误解
> - “PagedAttention 是新的注意力变体。” 不对，它主要是内存管理策略。
> - “只要模型支持 KV cache，就自动不需要分页。” 不对，大规模服务里页面管理往往决定吞吐。

## 例子或反例

多个不同长度请求并发生成时，若没有分页式管理，显存中会留下许多无法高效复用的空洞；有页式布局时，请求结束后更容易回收和复用页面。
