# MQA、GQA、MLA 如何做带宽折中

## 问题定义

为什么现代模型在推理优化时，不是优先砍 Query 头，而是优先压缩或共享 K/V？

## 直觉解释

Query 只活在当前步，K/V 要跟随整个历史反复被读取。真正昂贵的是历史 K/V 的存储和带宽，而不是当前 Q 的一次性计算。

## 形式化推导

- MHA：每个 Q 头都有对应 K/V 头，表达充分但 cache 大。
- MQA：多个 Q 头共享一组 K/V，最省带宽。
- GQA：把多个 Q 头分组共享 K/V，在质量与带宽之间折中。
- MLA：尝试把 K/V 压到更紧凑的潜表示中，再参与注意力计算。

## 工程意义

GQA 之所以成为很多现代 LLM 的默认，是因为它常能在不明显伤质量的前提下，大幅降低 KV cache 压力。MQA 更激进，MLA 更复杂，但目标都相同：减少解码时对历史状态的读取成本。

## 常见误解

> [!warning] 常见误解
> - “这些都只是工程 trick。” 不对。它们改变了表示瓶颈，是结构性改动。
> - “共享 K/V 不会影响质量。” 不对，只是影响可能可接受。

## 例子或反例

长上下文在线服务里，即使 FLOPs 还够，MHA 的 KV cache 也可能先把吞吐打穿；这时 GQA 往往比继续加头更有现实价值。
