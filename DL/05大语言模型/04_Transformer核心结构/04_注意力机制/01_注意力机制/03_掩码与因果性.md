# 掩码与因果性

> [!abstract]
> 掩码解决的是“谁允许看谁”的问题；因果性解决的是“当前时刻能不能偷看未来”的问题。

> [!warning]
> 因果 mask 不是位置编码。  
> 它只负责限制信息流，不负责告诉模型“第几个 token 在哪里”。

## 为什么注意力一定要配合 mask

如果没有 mask，全局注意力默认允许任意位置互相可见。  
这在很多场景下会直接出错：

- 批处理时，`<pad>` 不应该参与真实语义计算
- 自回归生成时，当前位置不能看到未来 token
- 条件生成时，源序列和目标序列的可见范围往往不同

所以 mask 的本质，是把“不允许发生的信息交互”在注意力打分阶段就裁掉。

## 三类最常见的 mask

| 类型 | 屏蔽对象 | 典型场景 | 作用 |
| --- | --- | --- | --- |
| Padding Mask | 填充位 `<pad>` | 批处理 | 防止无效位置污染注意力分布 |
| Causal Mask | 未来位置 | Decoder 自回归生成 | 保证当前位置只看过去和当前 |
| Cross-Attention Mask | 源端无效位置或受限区域 | Encoder-Decoder / 多模态 | 控制条件信息流范围 |

## 数学形式

令 mask 矩阵为 $M$，其中：

- $M_{ij} = 0$ 表示位置 $j$ 对查询位置 $i$ 可见
- $M_{ij} = -\infty$ 表示位置 $j$ 对查询位置 $i$ 不可见

则注意力可写为：

$$
\text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^\top}{\sqrt{d_k}} + M\right)V
$$

它的关键点是：**mask 加在 softmax 之前**。  
因为这样被屏蔽的位置在 softmax 后会接近 0，后续就不会参与加权求和。

## 因果 mask 长什么样

对长度为 4 的序列，典型下三角因果 mask 可以表示为：

| Query \ Key | 1 | 2 | 3 | 4 |
| --- | --- | --- | --- | --- |
| 1 | 0 | $-\infty$ | $-\infty$ | $-\infty$ |
| 2 | 0 | 0 | $-\infty$ | $-\infty$ |
| 3 | 0 | 0 | 0 | $-\infty$ |
| 4 | 0 | 0 | 0 | 0 |

这表示：

- 第 1 个位置只能看自己
- 第 2 个位置只能看 1、2
- 第 3 个位置只能看 1、2、3
- 第 4 个位置可以看 1 到 4

这就是“自回归”的形式化表达。

## 掩码约束的是信息流，不是顺序本身

很多初学者会把“因果 mask”与“顺序建模”混为一谈。要分清楚：

- 因果 mask 只保证未来位置不可见
- 位置编码才负责把顺序信息显式注入模型

也就是说，如果只有 causal mask、没有位置编码，模型虽然不能偷看未来，但它仍然很难知道“当前看到的几个 token 到底谁在前谁在后”。

## 训练与推理时的差别

### 训练阶段

- 常配合 Teacher Forcing 使用
- 目标序列整体右移后送入 Decoder
- 因果 mask 明确禁止当前位置看未来
- padding mask 用来处理批内长度不一致

### 推理阶段

- 模型逐 token 生成
- 已生成的历史 $K/V$ 会进入 KV cache
- 因为未来 token 根本还不存在，所以往往不需要显式构造完整未来 mask
- 但跨样本批处理时，padding mask 仍然有意义

## 工程实现要点

> [!info]
> 真正容易踩坑的，不是“有没有 mask”，而是 mask 的 dtype、shape 和广播方式是否和注意力实现一致。

- 布尔 mask 与加性 mask 都常见，但最终都要在 softmax 前生效。
- 半精度训练里，经常用 `-1e4` 或 `-1e9` 近似 `-\infty`，避免数值溢出。
- 常见张量形状是 `(B, 1, 1, L_k)`、`(B, 1, L_q, L_k)` 或 `(1, 1, L, L)`，具体取决于实现是否广播到 head 维。
- FlashAttention、xFormers 等高效实现通常对 mask 形状有更严格要求。

## 一句话区分三种相关概念

| 概念 | 它回答什么问题 |
| --- | --- |
| Position Encoding | 序列位置在哪里 |
| Attention Mask | 哪些位置允许交互 |
| KV Cache | 历史信息如何高效复用 |
