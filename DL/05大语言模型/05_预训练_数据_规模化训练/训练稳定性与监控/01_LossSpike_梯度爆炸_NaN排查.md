# Loss Spike, 梯度爆炸与 NaN 排查

训练大模型就像开飞机，Loss Spike（损失刺突）和 NaN（数值溢出）是常见的飞行事故。

## 1. Loss Spike (损失刺突)

Loss 突然剧烈上升，然后可能缓慢下降，或者直接发散。

### 1.1 常见原因
*   **脏数据**: 某个 Batch 中包含损坏的样本（如全是噪音、长度极长、标签错误）。
*   **学习率过大**: 步子迈大了，跳出了局部最优解。
*   **优化器状态异常**: Adam 的动量积累了错误的统计量。

### 1.2 语音训练中的特例
*   **CTC Loss 异常**: 当目标文本长度大于音频帧数时，CTC Loss 会变成无穷大 (Inf)。
*   **静音片段**: 纯静音片段在做 Log-Mel 时可能出现极小值，导致 Log 运算溢出。

### 1.3 解决方案
*   **数据清洗**: 严格过滤异常长度和异常 SNR 的音频。
*   **Gradient Clipping**: 限制梯度范数。
*   **回滚 (Rollback)**: 检测到 Spike 后，加载上一个 Checkpoint，跳过当前 Batch，减小学习率继续训练。

## 2. 梯度爆炸 (Gradient Exploding)

梯度值变得极大，导致权重更新幅度过大，模型参数瞬间“飞”出合理范围。

### 2.1 现象
*   Loss 变成 NaN。
*   权重参数变成 NaN。

### 2.2 语音模型特有的梯度问题
*   **RNN-T / LSTM**: 循环神经网络天然容易梯度爆炸（BPTT 时间步太长）。
*   **Deep Conformer**: 深层卷积结构在初始化不当时容易导致信号放大。

## 3. NaN (Not a Number) 排查指南

### 3.1 来源
*   **除以零**: 归一化层中方差为 0。
*   **Log(0)**: 输入概率为 0。
*   **Overflow**: FP16 精度下，数值超过 65504。

### 3.2 排查步骤
1.  **检查混合精度**: 切换到 BF16（几乎能解决 90% 的 Overflow 问题）。
2.  **检查 Loss 函数**: 确保 Softmax 前的 Logits 没有过大，CTC Loss 的输入长度合法。
3.  **Debug Mode**: 开启 PyTorch 的 `detect_anomaly`，定位产生 NaN 的具体算子。
