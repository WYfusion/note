## 概述

推理速度的本质是 **GEMM 计算 + KV cache 访存 + 自回归串行 + 调度开销** 的博弈。Prefill 偏 compute-bound，Decode 偏 memory-bound。

---

## 7.1 推理分两段

|阶段|处理内容|主导瓶颈|对应指标|
|---|---|---|---|
|**Prefill**|处理整段 prompt（一次前向所有 token）|Compute-bound（大矩阵 GEMM）|TTFT（Time To First Token）|
|**Decode**|自回归逐 token 生成|Memory-bound（KV 读取 + 权重读取）|TPOT（Time Per Output Token）|

```mermaid
graph LR
    A["Prompt tokens"] --> B["Prefill 阶段"]
    B --> C["生成 KV Cache"]
    C --> D["Decode 阶段"]
    D --> E["逐 token 生成"]
    E -->|"自回归"| D
```

---

## 7.2 单 token Decode 近似

### FLOPs 主项

$$c_{decode\_token} \approx 2N_{active} + c_{kv}(T)$$

- $2N_{active}$：本步所有线性层的矩阵-向量乘（Dense 即 $2N$，MoE 取 top-k expert）

- $c_{kv}(T) = 2 times T times h_{kv} times d_{head}$：对历史 KV 的注意力计算

---

## 7.3 延迟的工程近似

$$\text{latency/token} \approx \max(t_{compute},\ t_{memory}) + t_{scheduler}$$

其中：

$$t_{compute} \approx \frac{2N_{active}}{\text{sustained\_FLOPs}}$$

$$t_{memory} \approx \frac{\text{weight\_bytes} + \text{KV\_bytes\_per\_step}}{\text{memory\_bandwidth}}$$

### Arithmetic Intensity（算术强度）

$$AI = \frac{\text{FLOPs}}{\text{bytes\_loaded}}$$

- Decode 时 $AI$ 很低（batch=1 时约 1~2 FLOP/byte）→ **memory-bound**

- Prefill 时 $AI$ 较高 → **compute-bound**

> [!important]
> 
> **Roofline 模型**：当 $AI < AI_{ridge}$（ridge point = peak FLOPs / peak bandwidth）时，性能受限于带宽而非算力。H100 的 $AI_{ridge} \approx 312 / 3.35 \approx 93$ FLOP/byte。Decode batch=1 时 $AI approx 1$，远低于 ridge → **纯带宽瓶颈**。

---

## 7.4 推理速度的第一瓶颈

|场景|主瓶颈|优化方向|
|---|---|---|
|短上下文 + 小模型|Compute-bound|更快的 Tensor Core / 更高 FLOPs|
|长上下文 + 小 batch|KV memory-bound|GQA/MQA, KV 量化, FlashDecoding|
|大并发 + 长上下文|KV + 调度瓶颈|PagedAttention, continuous batching, KV 共享|

> [!important]
> 
> "降权重量化" **不一定** 解决长上下文 decode 瓶颈——真正关键常是 **KV cache + 带宽 + 调度**。

---

## 7.5 吞吐与时延的关系

- 增大 batch / continuous batching → 吞吐↑，但单请求尾延迟 **未必下降**

- 生产系统必须同时监控：

|指标|含义|优化倾向|
|---|---|---|
|TTFT|首 token 延迟|Prefill 效率 + 排队时间|
|TPOT|每输出 token 时间|Decode 效率 + KV 带宽|
|总吞吐 tok/s|系统级总生成速率|Batch 策略 + 并行度|
|P95 / P99 时延|尾延迟|调度公平性 + 预算管理|
