## 0. 定位

> 一句话：rollout 是 GRPO 最耗时环节，ms-swift 提供三种部署模式：**colocate**（与训练同卡）、**async**（独立 rollout server）、**hybrid**。

---

## 1. 三种部署模式对比

|模式|实现|优点|缺点|**Colocate**|训练 GPU 同时跑 vLLM|部署简单，不需额外卡|纯 rollout 时训练空转|
|---|---|---|---|---|---|---|---|
|**Async (Server)**|独立 GPU 跑 `swift rollout`|训练与采样并行|需额外 GPU，部署复杂|**Hybrid**|部分同卡 + 部分跨卡|灵活|调优复杂|

## 2. Colocate（入门首选）

```bash
# 4×A100：训练与 rollout 同卡
NPROC_PER_NODE=4 swift rlhf \
    --rlhf_type grpo \
    --model Qwen/Qwen2.5-7B-Instruct \
    --dataset AI-MO/NuminaMath-CoT \
    --use_vllm true \
    --vllm_mode colocate \
    --vllm_gpu_memory_utilization 0.5 \   # 训练估一半，vLLM 估一半
    --num_generations 8 \
    --train_type lora \
    --output_dir out/grpo-colocate
```

要点：

1. **显存应分配**：`vllm_gpu_memory_utilization` 不要超过 0.5，为训练留足够。

1. **训 rollout 串行**：rollout 时训练等待，训练时 vLLM 空转。

1. **适用**：GPU 资源有限（单机 4-8 卡）的启动阶段。

## 3. Async 部署

```bash
# === Step 1: 启动 rollout server（8 张 GPU） ===
CUDA_VISIBLE_DEVICES=0,1,2,3,4,5,6,7 \
swift rollout \
    --model Qwen/Qwen2.5-7B-Instruct \
    --infer_backend vllm \
    --tensor_parallel_size 8 \
    --max_num_seqs 256 \
    --gpu_memory_utilization 0.9 \
    --port 8088

# === Step 2: 启动训练（8 张另外 GPU） ===
CUDA_VISIBLE_DEVICES=8,9,10,11,12,13,14,15 \
NPROC_PER_NODE=8 swift rlhf \
    --rlhf_type grpo \
    --model Qwen/Qwen2.5-7B-Instruct \
    --use_vllm true \
    --vllm_mode server \
    --vllm_server_url http://localhost:8088 \
    --num_generations 8 \
    --output_dir out/grpo-async
```

## 5. 权重同步机制

Rollout server 使用的策略与训练 policy 会逐渐偏离，需要定期同步。ms-swift 提供两种：

|模式|参数|说明|
|---|---|---|
|推送 LoRA|`vllm_sync_lora_only=true`|仅同步 LoRA 权重，几乎免费|

## 7. 性能参考

> [!important]
> 
> **经验**：在 Qwen-7B + 8×H100 上，async 模式相比 colocate 快约 1.6-2.0x，代价是多需 8 张 GPU。如果只有一机机友资源，colocate 加 LoRA 是最环保的选择。

> [!important]
> 
> Async 模式下 rollout 与训练跨轮，可能会出现**同一 prompt 不同轮采样不一致**（policy 已变）。必须配合 importance sampling correction，ms-swift 默认已处理。
