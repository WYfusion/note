## 0. 定位

> 一句话：本页讲清 **GRPO 家族**——以 DeepSeek-R1 引爆的「**可验证奖励 RL**」范式，从 GRPO 原型到 DAPO/GSPO/SAPO/CISPO/CHORD 的工程演进，是 ms-swift 4.x 的旗舰能力。

---

## 1. 为什么需要 GRPO 家族

经典 PPO 的两大工程痛点：

1. **需要 critic 网络**：再训一个与 policy 同尺寸的 value model，**显存翻倍**。

2. **优势估计不稳**：GAE 在长 rollout 上方差大。

GRPO（**G**roup **R**elative **P**olicy **O**ptimization）的核心创新：

> [!important]
> 
> 对每个 prompt **采样 G 条响应**，用「**组内归一化**」替代 critic 估计 advantage——彻底**去掉 value model**，节省一半显存。

$$A_{i,t} = \frac{r_i - \text{mean}(\{r_1, \dots, r_G\})}{\text{std}(\{r_1, \dots, r_G\})}$$

其中 $r_i$ 是第 $i$ 条 rollout 的总奖励。

---

## 2. GRPO 家族版本对比

|算法|提出方|核心改进|解决什么问题|
|---|---|---|---|
|**GRPO**|DeepSeek (DeepSeek-R1)|组内归一化 advantage、去 critic|显存与方差|
|**DAPO**|ByteDance|Clip Higher / Dynamic Sampling / Token-level Loss / Overlong Filtering|熵坍缩、全错样本浪费、超长惩罚|
|**GSPO**|Qwen 团队|**序列级**重要性采样替代 token 级|MoE 训练不稳定|
|**CISPO**|—|修正 IS 偏差|off-policy 偏移|
|**SAPO / CHORD**|—|采样策略调整|多样性|
|**RLOO**|Cohere|留一法 baseline|简化 GRPO|
|**Reinforce++**|—|Reinforce + KL + GAE|简化 PPO|

## 3. ms-swift GRPO 核心命令

ms-swift GRPO 训练有 **Colocate**（训练与推理共享显卡）和 **Server**（推理服务独立部署）两种模式：

```bash
# Colocate 模式：4 卡训 + 4 卡推理共享 8 张 GPU
CUDA_VISIBLE_DEVICES=0,1,2,3,4,5,6,7 \
NPROC_PER_NODE=8 swift rlhf \
    --rlhf_type grpo \
    --model Qwen/Qwen3-8B \
    --dataset AI-MO/NuminaMath-CoT \
    --reward_funcs accuracy format \
    --use_vllm true \
    --vllm_mode colocate \
    --num_generations 8 \
    --max_completion_length 4096 \
    --temperature 1.0 \
    --beta 0.04 \
    --learning_rate 1e-6 \
    --gradient_accumulation_steps 4

# 升级到 DAPO：开启 Clip Higher 与 Dynamic Sampling
swift rlhf --rlhf_type grpo \
    --model Qwen/Qwen3-8B \
    --dataset AI-MO/NuminaMath-CoT \
    --reward_funcs accuracy format \
    --epsilon 0.2 --epsilon_high 0.28 \
    --dynamic_sample true \
    --overlong_filter true \
    --loss_type token_mean \
    --num_generations 16
```

> [!important]
> 
> **MoE 训练务必用 GSPO**：研究表明 GRPO 在 MoE 上由于 token 级 IS 比例方差过大会发散，**GSPO 改用序列级 IS 比例显著稳定训练**。

## 4. 自定义 Reward Function（GRPO 灵魂）

GRPO 的工程灵魂是「**奖励函数即任务定义**」。ms-swift 提供插件机制：

```python
# my_reward.py - 自定义奖励：判断输出是否包含正确答案
import re
from swift.plugin.orm import ORM, orms

class MathAccuracyReward(ORM):
    """奖励函数：从模型输出抽取最终答案，与 ground truth 对比。"""
    
    def __call__(self, completions, solution, **kwargs):
        """
        completions: List[str]  # 模型本批次的所有 rollout
        solution:    List[str]  # 对应每条 rollout 的标准答案
        return:      List[float] # 每条 rollout 的奖励
        """
        rewards = []
        for comp, sol in zip(completions, solution):
            # 提取 \boxed{...} 中的答案
            match = re.search(r'\\boxed\{(.+?)\}', comp)
            pred = match.group(1).strip() if match else None
            # 严格比对：完全匹配得 1.0，否则 0.0
            rewards.append(1.0 if pred == sol.strip() else 0.0)
        return rewards

# 注册到 ms-swift
orms['math_acc'] = MathAccuracyReward
```

命令行启用：

```bash
swift rlhf --rlhf_type grpo \
    --external_plugins my_reward.py \
    --reward_funcs math_acc format \
    ...
```

> [!important]
> 
> **多奖励组合的工程原则**：先「**格式奖励**」让模型输出可解析，再「**结果奖励**」推动正确率。多奖励**等权相加**或加权求和，避免梯度被某个 reward 主导。

## 6. 工程判断：何时用 GRPO

> [!important]
> 
> **GRPO 适用判据**：(1) 任务有**可程序化验证**的奖励（数学答案正确、代码执行通过、格式正则匹配）；(2) 已有强 SFT base；(3) 计算预算允许 G≥8 的 rollout。

> [!important]
> 
> **反例**：开放式生成（写诗、对话）**不要用 GRPO**。没有客观奖励信号时，GRPO 会让模型坍缩到 reward hack 模式。
