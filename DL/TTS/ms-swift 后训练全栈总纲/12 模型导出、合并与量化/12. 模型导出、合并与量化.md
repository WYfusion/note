## 2. LoRA 合并：Merge or Not

|选择|优点|缺点|
|---|---|---|
|**Merge**|推理零额外开销；可量化|丢失多任务切换能力|
|**保留 Adapter**|快速切换多个任务适配器|vLLM/SGLang 加载 adapter 有限制|

```bash
# Merge LoRA 为完整模型
swift export --model Qwen/Qwen3-8B \
    --adapters output/sft/checkpoint-xxx \
    --merge_lora true \
    --output_dir merged_model
```

> [!important]
> 
> **常见坑**：QLoRA 训练后的 adapter 在 merge 时必须**先反量化 base** 再 merge，否则精度损失严重。ms-swift `--merge_lora true` 会自动处理。

## 3. 量化方法全景

|方法|位宽|需校准集|部署后端|推荐场景|
|---|---|---|---|---|
|**AWQ**|4 (W4A16)|✅ 少量|vLLM / LMDeploy|生产首选、质量佳|
|**GPTQ**|4 / 8|✅ 少量|vLLM|生态成熟|
|**FP8**|8|❌|vLLM (H100/H200)|新 GPU 小损失高吞吐|
|**HQQ**|2/3/4/8|❌|transformers|零 calib、快量化|
|**BNB (NF4)**|4|❌|transformers|QLoRA 训练默认|
|**AQLM**|2|✅ 大量|transformers|极限压缩|
|**EETQ**|8|❌|transformers|快速 int8|

## 4. AWQ 量化试一路

```bash
# Step 1: 准备校准集（使用训练集同分布 ~256 条）
# Step 2: AWQ 量化
swift export --model merged_model \
    --quant_method awq \
    --quant_bits 4 \
    --dataset calib_data.jsonl \
    --quant_n_samples 256 \
    --output_dir merged_model-awq

# Step 3: 部署量化后模型
swift deploy --model merged_model-awq \
    --infer_backend vllm \
    --served_model_name my-model-awq
```

> [!important]
> 
> **量化后必须重评**：使用与训练一致的 benchmark 重测，常见 1-3% 损失是可接受的，超过 5% 说明校准集不匹配。

---

## 5. FP8 量化（H100/H200）

```bash
swift export --model merged_model \
    --quant_method fp8 \
    --output_dir merged_model-fp8
```

> [!important]
> 
> **为什么 FP8 在新 GPU 上赢**：Hopper / Blackwell 架构**原生支持 FP8 矩阵运算**，FLOPs 是 bf16 的 2x。与 int8 不同，无需 dequant，在 H100+ 上逼近 fp16 质量，同时吞吐与 int8 相当。

## 6. 推送到 ModelScope / HuggingFace

```bash
swift export --model merged_model-awq \
    --push_to_hub true \
    --hub_model_id my-org/my-model-awq \
    --hub_token $HUB_TOKEN
```
