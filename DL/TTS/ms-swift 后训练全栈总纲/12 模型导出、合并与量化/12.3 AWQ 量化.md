## 0. 定位

> 一句话：AWQ = Activation-aware Weight Quantization，用**重要性加权**保护关键权重。量化快，推理也快。

---

## 1. 核心思路

GPTQ 逐列补偿费时。AWQ 不补偿。代之以不同思路：**重要权重不量化在 "原始位"**，而是用激活文位面重要性估计。

```jsx
设某个线性层 y = Wx。重要权重 ≈ 列被常用激活（|x_i| 大）。
AWQ:
  1. 用校准集计算每列的平均幅度 s_i = mean(|x_i|)
  2. 缩放 W 与 x: W' = W · diag(s)^α, x' = diag(s)^-α · x
  3. 量化 W' (重要列变大，量化误差低)
```

## 2. ms-swift 命令

```bash
swift export \
    --model out/qwen-sft \
    --quant_method awq \
    --quant_bits 4 \
    --quant_dataset my-calib-prompts \
    --quant_n_samples 256 \
    --quant_seqlen 2048 \
    --output_dir merged/qwen-7b-awq
```

## 3. AWQ 与 GPTQ 对比

|项|GPTQ|AWQ|原理|逐列 OBQ 补偿|列重要性缩放|
|---|---|---|---|---|---|
|量化耗时|慢 (·1×)|快 (·0.3×)|质量|极高|高|
|推理加速|快 (Marlin kernel)|更快 (vLLM AWQ kernel)|量化集需求|1024+|256-512 即可|

## 4. 校准集

```bash
# 生产采样为首选
swift export ... \
    --quant_dataset prod_samples.jsonl \   # 你的生产数据
    --quant_n_samples 256
```

AWQ 对样本数不敏感，256 已足在多数场景。

## 5. 部署

```bash
swift deploy \
    --model merged/qwen-7b-awq \
    --infer_backend vllm \
    --quantization awq_marlin \
    --max_model_len 8192 \
    --gpu_memory_utilization 0.9
```

## 6. 质量验证

```python
from swift.llm import infer_main, InferArguments

# 原始与量化在同一批 prompt 上对比
prompts = [
    '请解释什么是 Notion?',
    '写一个 Python 函数计算阶乘',
    '总结 transformer 架构的主要组件',
]
orig = infer_main(InferArguments(model='out/qwen-sft', infer_backend='vllm'), prompts)
awq  = infer_main(InferArguments(model='merged/qwen-7b-awq', infer_backend='vllm'), prompts)
for o, a in zip(orig, awq):
    print('---')
    print('orig:', o[:200])
    print('awq :', a[:200])
```

## 7. 踩坑提示

> [!important]
> 
> - **量化前务必合并 LoRA**：不能直接量化 LoRA adapter。
> 
> - **AWQ scale group_size**：默认 128，一般不需调。
> 
> - **校准集多样性**：多领域混合会使量化更鲁棒。
> 
> - **部署后端**：vLLM、SGLang、LMDeploy 都支持 AWQ。LMDeploy 原生提高 W4A16 推理。
