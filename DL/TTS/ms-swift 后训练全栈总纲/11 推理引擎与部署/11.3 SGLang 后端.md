## 0. 定位

> 一句话：SGLang = LMSys 开发的结构化生成推理引擎，核心是 **RadixAttention**（多轮共享前缀）。

---

## 1. 核心机制：RadixAttention

多轮对话中，同一 system prompt 会被不同请求重复处理。SGLang 用前缀树缓存这些 prefix 的 KV：

![[11.3 SGLang 后端 - 1. 核心机制：RadixAttention.excalidraw|800]]

效果：在 Agent / RAG 场景下、多轮同一 system prompt 多个用户请求上，吞吐提高 1.5-3x。

## 2. 启动服务

```bash
swift deploy \
    --model out/qwen-sft \
    --infer_backend sglang \
    --port 8000 \
    --max_model_len 8192 \
    --tensor_parallel_size 2 \
    --mem_fraction_static 0.85 \
    --max_running_requests 256
```

## 3. 常用参数

|参数|含义|
|---|---|
|`max_running_requests`|最大并发请求|
|`schedule_policy`|fcfs / lpm (Longest Prefix Match)|

## 4. 何时优先 SGLang

|场景|增益|
|---|---|
|**RAG**|检索 prompt 重复多|
|**多用户同 system**|同个 system prompt 被 N 人请求|

## 5. 与 vLLM 的性能对比

```jsx
Qwen2.5-7B + 1×A100 80G + 1024 ShareGPT 请求
                       throughput（tok/s）  TTFT (avg)
  vLLM:                ~2500              80ms
  SGLang (同 system):  ~3500              50ms
  SGLang (不共享):       ~2400             80ms
```

在 prefix 不共享场景下二者接近，共享多时 SGLang 胜出。

## 6. 调用

```python
# OpenAI 兼容接口与 vLLM 一致
from openai import OpenAI
client = OpenAI(base_url='http://localhost:8000/v1', api_key='dummy')
resp = client.chat.completions.create(
    model='my-qwen',
    messages=[
        {'role':'system','content':'You are a helpful assistant.'},
        {'role':'user',  'content':'北京天气怎么样?'},
    ],
)
print(resp.choices[0].message.content)
```

## 7. 踩坑提示

> [!important]
> 
> - **VL / MoE 支持与 vLLM 有差异**：部分 VL 模型仅 vLLM 支持，部分 MoE 仅 SGLang。
> 
> - **RadixAttention 需多请求**：单请求场景下增益 ≈ 0。
> 
> - **启动参数差异**： SGLang 参数名不同于 vLLM (mem_fraction_static 不是 gpu_memory_utilization)。
