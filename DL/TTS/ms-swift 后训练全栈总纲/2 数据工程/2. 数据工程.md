## 0. 定位

> 一句话：本页给出 ms-swift **各类训练任务**（SFT/DPO/RM/KTO/GRPO/MLLM/Agent/Embedding）的**统一数据 schema 与工程规范**，并明确「数据优先级 / 清洗流程 / 污染检测 / 合成数据」四条主线的工程判断。

---

## 1. 数据优先级铁律

> [!important]
> 
> **优先级链**：真实业务数据（最高）→ 高质量人工标注 → 强模型合成数据 → 开源通用数据（最低）。当训练目标是「业务效果」而非「学术 benchmark」时，**1 万条业务数据 > 10 万条开源数据**。

---

## 2. 各任务数据 schema 对照表

|任务|核心字段|典型示例|
|---|---|---|
|**CPT**|`text`|`{"text": "长篇领域文档..."}`|
|**SFT**|`messages`|`{"messages": [{"role": "system", ...}, {"role": "user", ...}, {"role": "assistant", ...}]}`|
|**DPO / CPO / SimPO / ORPO**|`messages` • `rejected_response`|messages 末尾 assistant 即 chosen，另带 rejected|
|**KTO**|`messages` • `label` (bool)|`{"messages": [...], "label": true}` 单边偏好|
|**RM**|`messages` • `rejected_response`|同 DPO 格式（pairwise）|
|**GRPO / PPO**|`messages` (仅 prompt) + 可选 `solution`|`{"messages": [...user...], "solution": "42"}`|
|**多模态 SFT**|`messages` • `images`/`videos`/`audios`|messages 中含 `<image>` 占位符|
|**Agent**|`messages` • `tools`|`{"messages": [...], "tools": [{"type":"function", ...}]}`|
|**Embedding**|`query` • `positive` • `negative`|InfoNCE 三元组|
|**Reranker**|同 Embedding，pairwise 或 listwise|—|

---

## 3. SFT messages 标准格式（最核心）

ms-swift 的所有训练数据最终都向 `messages` 格式靠拢，因为它**与 OpenAI Chat API 兼容**，且能**统一多轮、工具调用、多模态**：

```json
{
  "messages": [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "What is 2+2?"},
    {"role": "assistant", "content": "4"},
    {"role": "user", "content": "Why?"},
    {"role": "assistant", "content": "Because addition..."}
  ]
}
```

**关键工程点**：

- ✅ 多轮训练时，**所有 assistant 段都参与 loss**（默认行为）

- ✅ 仅最后一轮 assistant 算 loss：用 `--loss_scale last_round`

- ✅ system 设计影响巨大：**长 system 在 packing 时显著占位**，建议 < 200 token

## 5. 数据加载与预处理代码示例

用 ms-swift Python API 加载并预览数据：

```python
# 从本地 JSONL 加载 SFT 数据，按 Qwen3 template 编码
from swift.llm import get_model_tokenizer, get_template, get_dataset

MODEL = 'Qwen/Qwen3-8B'

# 1. 加载 tokenizer 和 template
_, tokenizer = get_model_tokenizer(MODEL, load_model=False)
template = get_template('qwen3', tokenizer, max_length=2048)

# 2. 加载数据集（支持 ModelScope id / HF id / 本地 jsonl）
train_ds, val_ds = get_dataset(
    datasets=['./data/my_sft.jsonl'],
    val_dataset=['./data/my_val.jsonl'],
    template=template,
    num_proc=4,             # 多进程加速
    streaming=False,        # 大数据集可设为 True 走流式
    strict=False,           # 容忍部分坏样本
)

# 3. 预览首条编码后样本
print(tokenizer.decode(train_ds[0]['input_ids']))
print('labels (loss mask):', train_ds[0]['labels'][:50])  # -100 表示不计 loss
print('seq len:', len(train_ds[0]['input_ids']))
```

> [!important]
> 
> `labels` 中值为 `-100` 的位置就是 **loss mask**，对应 system + user + 特殊 token，不参与梯度计算。这是 SFT「只学 assistant」的实现关键。

## 6. 工程判断：何时合成数据

|场景|建议|原因|
|---|---|---|
|业务数据 < 1k 条|合成 + 人工精校|纯合成质量不足，纯人工成本高|
|业务数据 1k - 10k|SFT 业务 + DPO 合成|SFT 锚定能力，DPO 用合成偏好对齐风格|
|业务数据 > 10k|主用业务，合成补长尾|业务数据已能锚定，合成数据降低过拟合|
|开放式生成|避免纯合成 SFT|合成数据风格趋同，模型容易丧失多样性|
