## 3. 12 大易错点（必背）

|#|易错点|规避|
|---|---|---|
|2|只微调 LLM，忽略 ASR 错误级联|优化 ASR 热词 + ITN，再调 LLM|
|4|TTS 数据授权不清|合规审查 + 授权文件留档|
|6|训练集和评测集来自同一说话人|评测集分层冻结，跨说话人|
|8|没有热词机制，专有名词错误率高|维护热词表，定期更新|
|10|TTS 只看自然度，不看相似度 / 稳定性|多指标并查（MOS+SMOS+稳定性）|
|12|多模态数据中 audio 占位符与文件不一致|数据校验脚本强制校验|

## 4. 上线 Checklist

```
□ 数据
  □ 采样率 / 声道统一
  □ 评测集已冻结，跨说话人
  □ 占位符与音频文件一一对应
  □ 授权文件齐全

□ 训练
  □ 选择正确冻结策略（默认 freeze_vit=true, freeze_aligner=false）
  □ Speech : Text 数据混合（默认 2:1）
  □ 小样本冒烟通过
  □ 中间 checkpoint 评测通过

□ 评测
  □ ASR 分场景 / 口音 / 噪声 CER
  □ LLM 意图 / 工具调用准确率
  □ TTS MOS / 相似度 / 稳定性
  □ 端到端业务完成率 + P95 延迟

□ 部署
  □ ASR / LLM / TTS 独立扩缩容
  □ 流式 / 非流式按场景选择
  □ 监控 P50/P95/P99 + 失败率
  □ 日志回流（保留 raw audio）
  □ 灰度 5-10%，固定声学层只切 LLM

□ 异常处理
  □ 噪声 / 静音 / 打断 / 重听状态机
  □ ASR 失败兜底（重试 / 转人工）
  □ LLM 拒答 / 安全策略
  □ TTS 异常字符回退处理
```

## 5. 流式压测脚本（示意）

```python
import asyncio, time, statistics
import aiohttp

async def one_session(session, audio_chunks):
    t0 = time.time()
    ttft = None
    last_t = t0
    deltas = []
    async with session.ws_connect('ws://gateway/voice') as ws:
        for ch in audio_chunks:
            await ws.send_bytes(ch)
        async for msg in ws:
            now = time.time()
            if ttft is None:
                ttft = now - t0
            deltas.append(now - last_t)
            last_t = now
            if msg.data == b'<END>':
                break
    return ttft, sum(deltas), len(deltas)

async def main(n_concurrent=64):
    chunks = load_audio_as_chunks('test.wav', chunk_ms=200)
    async with aiohttp.ClientSession() as s:
        tasks = [one_session(s, chunks) for _ in range(n_concurrent)]
        results = await asyncio.gather(*tasks)
    ttfts = [r[0] for r in results]
    durs = [r[1] for r in results]
    print(f'TTFT  P50={statistics.median(ttfts)*1000:.0f}ms  P95={sorted(ttfts)[int(0.95*len(ttfts))]*1000:.0f}ms')
    print(f'TOTAL P50={statistics.median(durs)*1000:.0f}ms  P95={sorted(durs)[int(0.95*len(durs))]*1000:.0f}ms')

asyncio.run(main())
```

## 6. 落地路线总结（再强调）

|诉求|路线|
|---|---|
|**工业稳定优先**|VAD → ASR → ms-swift LLM (SFT/DPO/GRPO) → TTS → 流式部署|
|**体验自然优先**|Speech LLM/Omni → ms-swift SFT → 端到端部署|
|**成本可控优先**|专用 ASR/TTS 不动，只用 ms-swift 微调中间 LLM|
|**领域知识优先**|ASR 热词 / 纠错 → LLM CPT/SFT → RAG → TTS|
|**推理能力优先**|Speech/Text SFT → GRPO → 业务验证器评测|

## 7. 关键主张总结

> [!important]
> 
> 1. **ASR/TTS 是声学与语音工程问题。**
> 
> 1. **LLM 是语义、推理、对话、工具调用问题。**
> 
> 1. **ms-swift 的优势在语音多模态大模型训练与对齐。**
> 
> 1. **生产系统通常 = ASR/TTS 专用模型 + ms-swift 微调 LLM。**
> 
> 1. **端到端 Speech LLM 适合追求自然交互，但成本、评测、部署复杂度更高。**

> [!important]
> 
> **不要在没有评测能力的情况下上端到端语音模型** — 出问题难以归因，会拖死项目。
