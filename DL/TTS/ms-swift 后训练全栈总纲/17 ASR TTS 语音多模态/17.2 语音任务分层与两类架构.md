## 0. 定位

> 一句话：把语音系统拆成 6 层，再选 **级联** 还是 **端到端 Speech LLM** 两类架构之一。

---

## 1. 任务分层

|层级|任务|典型能力|
|---|---|---|
|**ASR**|语音识别|语音转文字、时间戳、热词、标点恢复|
|**Dialogue LLM**|对话决策|指令跟随、工具调用、RAG、业务推理|
|**Speech-to-Speech**|端到端语音交互|语音输入、语音输出、低延迟对话|

## 2. 级联系统

![[17.2 语音任务分层与两类架构 - 2. 级联系统 - 图 01.excalidraw|800]]

|优点|缺点|
|---|---|
|稳定可控、易评测、易上线、模块独立迭代|误差级联、延迟较高、丢失语气 / 情绪 / 重音|

## 3. 端到端 Speech LLM

![[17.2 语音任务分层与两类架构 - 3. 端到端 Speech LLM - 图 02.excalidraw|800]]

|优点|缺点|
|---|---|
|交互自然、保留语气情绪、统一训练、跨模态融合|成本高、评测难、部署复杂、流式实现难|

## 4. 两类架构对比

|维度|级联|端到端 Speech LLM|
|---|---|---|
|训练复杂度|低（每段独立）|高（联合）|
|数据需求|中（每段各自）|高（speech-text pair）|
|语气 / 情绪|丢失|保留|
|评测|WER / MOS / 业务指标分别评|需端到端 + 子模块|
|延迟|累加（>1s 常见）|可至 < 500ms（streaming）|
|追责定位|明确|困难|
|工业落地|主流|新兴|

## 5. ms-swift 在两类架构里的角色

```python
# 场景 A: 级联中只训中间 LLM（最常见）
# - 输入: ASR 后的文本 + 上下文
# - 输出: 文本回复
# - 训练: 普通 SFT, 数据是 (asr_text, response)

# 场景 B: 端到端 Speech LLM SFT
# - 输入: 音频 + 文本 prompt
# - 输出: 文本（或音频 token）
# - 训练: 语音 SFT, 数据是 messages with audio

speech_llm_sample = {
    'messages': [
        {'role': 'system', 'content': 'You are a voice assistant.'},
        {'role': 'user', 'content': '<audio>请帮我总结这段会议</audio>'},
        {'role': 'assistant', 'content': '本次会议主要讨论了三个议题……'},
    ],
    'audios': ['/path/to/meeting.wav'],
}

cascade_sample = {
    'messages': [
        {'role': 'system', 'content': 'You are a voice assistant.'},
        {'role': 'user', 'content': '请帮我总结这段会议（ASR 转写后的文本）'},
        {'role': 'assistant', 'content': '本次会议主要讨论了三个议题……'},
    ],
}
```

## 7. 工程权衡

> [!important]
> 
> **"先级联，再 Speech LLM"** 通常是稳健路径：
> 
> 1. 用级联快速跑通业务、积累 (asr_text, response) 数据。
> 
> 1. 评估业务完成率瓶颈在哪：ASR? LLM? TTS?
> 
> 1. 若瓶颈在语气情绪丢失（如客服共情） → 升级到 Speech LLM。
> 
> 1. 否则继续优化级联各段。

> [!important]
> 
> **易错**：上来就追端到端 Speech LLM，导致评测难、回归难、定位难，进度严重落后。
