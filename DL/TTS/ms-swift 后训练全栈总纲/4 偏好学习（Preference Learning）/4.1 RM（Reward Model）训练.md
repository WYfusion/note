## 0. 定位

> 一句话：Reward Model = 学一个 **打分器** $r_\phi(x, y)$，让人类偏好的回答得分高于不偏好的，后续 PPO/RLHF 用它提供梯度信号。

---

## 1. 何时需要 RM

|场景|是否需要 RM|经典 RLHF（PPO）|✅ 必需|
|---|---|---|---|
|DPO/SimPO/ORPO/CPO|❌ 直接从 pairwise 数据学习|GRPO（可验证）|❌ reward 来自规则函数|
|GRPO（主观任务）|✅ 可用 RM 作为软 reward|Best-of-N 采样 / 拒绝采样|✅ RM 打分选择|

## 2. 损失函数（Bradley-Terry）

给定成对样本 $(x, y_w, y_l)$（$y_w$ 优于 $y_l$）：

$$\mathcal{L}_{\text{RM}} = -\mathbb{E}_{(x,y_w,y_l)\sim \mathcal{D}} \Big[ \log \sigma\big( r_\phi(x, y_w) - r_\phi(x, y_l) \big) \Big]$$

本质上是对打分差做二分类（logistic）。

## 3. RM 架构

![[4.1 RM（Reward Model）训练 - 3. RM 架构.excalidraw|800]]

本质上是把 base LM 的 lm_head 换成一个输出 1 维的线性层，取最后 token（或 EOS）的隐藏状态计算。

## 6. 手写 RM Loss（理解用）

```python
import torch, torch.nn.functional as F

def rm_loss(model, batch):
    """batch 包含 chosen_input_ids / rejected_input_ids 及 attention_mask."""
    # forward 两次，分别得到 chosen / rejected 的打分
    chosen_logits = model(
        input_ids=batch['chosen_input_ids'],
        attention_mask=batch['chosen_attention_mask']).logits  # [B, T, 1]
    rejected_logits = model(
        input_ids=batch['rejected_input_ids'],
        attention_mask=batch['rejected_attention_mask']).logits
    
    # 取有效序列最后一个位置的打分
    chosen_lengths = batch['chosen_attention_mask'].sum(dim=1) - 1
    rejected_lengths = batch['rejected_attention_mask'].sum(dim=1) - 1
    
    chosen_rewards   = chosen_logits[range(len(chosen_lengths)),
                                     chosen_lengths].squeeze(-1)
    rejected_rewards = rejected_logits[range(len(rejected_lengths)),
                                       rejected_lengths].squeeze(-1)
    
    # Bradley-Terry loss
    loss = -F.logsigmoid(chosen_rewards - rejected_rewards).mean()
    
    # 准确率（训练中监控）
    accuracy = (chosen_rewards > rejected_rewards).float().mean()
    return loss, accuracy
```

ms-swift `--rlhf_type rm` 内部使用类似实现，上面代码仅供理解。

## 7. RM 质量评估

|指标|含义|可接受|**Pairwise Accuracy**|$P(r(y_w) > r(y_l))$ 在验证集上|> 65%|
|---|---|---|---|---|---|
|**RewardBench**|三方评测（Chat / Reasoning / Safety）|> 70%|**过打分检测**|SFT 样本该得 ≈ 0，跟 chosen 接近|避免 reward hacking|

> [!important]
> 
> - **RM 是 RLHF 最脆弱环节**：质量差会在 PPO 如装未装。  
>     - **避免过拟合**：1 epoch 足够，数据应多样化。  
>     - **记录平均 reward**：限制在 [-2, 2] 内帮助 PPO 稳定。
